# Squad Protocol

## How Agents Discover Their Squad

When the orchestrator spawns a sub-agent, it includes squad context in the launch prompt:

1. Read `.claude/squads/organization.yaml` for org-level governance
2. Identify which squad owns the target repo/domain
3. Load the squad YAML to get: skills, governance, metric targets
4. Pass to sub-agent: squad name, applicable skills, coverage requirements, constitutional gates

### Repo-to-Squad Mapping

Each squad YAML declares `spec.repos`. The orchestrator resolves which squad owns a file by matching its path against squad repo lists. If a file is not owned by any squad, it falls to the infra-team (catch-all).

### Skill Loading

Squad YAML declares `spec.skills`. When an agent works on a squad's repo, the orchestrator pre-loads those skills into the sub-agent's context (following the existing skill-registry protocol).

## How Manager Agents Evaluate Performance

The manager agent (ManagerAgent) runs evaluation via `/squad-report`:

1. **Collect**: Read skill-metrics.jsonl, error-learning.jsonl, active-tasks.json
2. **Compute**: Calculate per-agent and per-squad metrics
3. **Compare**: Check against thresholds in squad YAML and ManagerAgent config
4. **Propose**: Generate reconfiguration proposals if thresholds are breached
5. **Report**: Output a structured squad performance report

### Evaluation Schedule

| Trigger | Action |
|---------|--------|
| Weekly (Monday 9am) | Full `/retrospective` across all squads |
| After major incident | Targeted `/squad-report {affected-squad}` |
| On demand | User runs `/squad-report` or `/retrospective` |
| Budget alert (>80% limit) | Auto-trigger cost analysis |

## When Auto-Reconfiguration Triggers

Auto-reconfiguration is proposed (not applied) when:

| Condition | Action | Requires Approval |
|-----------|--------|-------------------|
| successRate < 0.80 | upgradeModel | Yes |
| architectureCompliance < 0.90 | retrainSkills | No (skills only) |
| issueResolutionTime > target | scaleUp | Yes |
| bugsIntroduced > 3 per week | addQAAgent | Yes |
| costPerTask > $2.00 | modelRotation | No (cost saving) |
| persistent underperformance (2+ weeks) | restructureSquad | Yes (manager approval) |

### Cooldown Rules

- Minimum 24h between reconfigurations per squad
- Maximum 3 reconfigurations per week per squad
- `restructureSquad` has a 7-day cooldown
- Cooldown state is tracked in Engram: `squad-reconfig/{squad-name}/cooldown`

### Applying Reconfigurations

1. Proposal is generated by `/squad-report` or `/retrospective`
2. Proposal is shown to the user for approval
3. If approved, the orchestrator updates the squad YAML
4. Changes take effect on next agent spawn for that squad
5. The reconfiguration is logged to Engram for future effectiveness tracking

## Escalation Policy

```
Agent detects issue
    |
    v
Agent attempts self-resolution (within autonomy.mode)
    |
    v (if unresolved)
Escalate to Squad Manager (ManagerAgent)
    |
    v (if manager cannot resolve)
Escalate to Organization level
    |
    v (if org-level issue)
Escalate to Human (user notification)
```

### Escalation Levels

| Level | Actor | Can Do | Cannot Do |
|-------|-------|--------|-----------|
| 1. Agent | Individual agent | Fix within own scope, retry | Change model, add agents |
| 2. Squad Manager | ManagerAgent | Propose reconfig, reassign tasks | Override constitutional gates |
| 3. Organization | Cross-squad coordination | Propose budget reallocation, cross-squad moves | Bypass human approval for destructive actions |
| 4. Human | User | Everything | N/A |

### Escalation Rules

- Level 1 -> 2: After 2 failed attempts or timeout > 2x expected
- Level 2 -> 3: When issue spans multiple squads or requires budget change
- Level 3 -> 4: Always for destructive actions, production changes, financial operations
- Skip to Level 4: Any security incident or constitutional gate violation

## Squad Membership Rules

- An agent can belong to multiple squads (e.g., sre-agent in payments + infra)
- Each squad has exactly one manager (ManagerAgent reference)
- Manager agents are shared across squads (single engineering-manager-agent evaluates all)
- Agent `count` field allows multiple instances of the same agent type
- Adding/removing squad members requires manager approval

## Integration with Existing Protocols

| Protocol | Integration |
|----------|-------------|
| SRE Protocol | SRE agents operate within their squad's scope but have cross-squad visibility for infra issues |
| Error Learning | Errors are attributed to the squad that owns the affected repo |
| Skill Adaptation | Skill feedback is scoped to the squad's skill set |
| Agent KPIs | Squad metrics feed into org-level KPIs |
| Fault Tolerance | Squad manager monitors agent health via active-tasks.json |
| Constitutional Gates | Squad YAML declares which gates apply to its domain |
| Model Routing | Squad manager can override model-routing for its agents |
