# Research — ADR-067 Phase 2: Defense-in-Depth Extension to rules/, hooks/, ADRs/

**Type**: Research-only (per ADR-069)
**Decision Status**: Awaiting operator triage
**Implementation NOT performed**
**Date**: 2026-04-24

## TL;DR

ADR-067 Phase 1 established a 3-layer pattern (template + PostToolUse hook + audit
test) for `skills/*/SKILL.md`. Phase 2 extends that pattern to three new artifact
types. The highest-leverage target is `docs/adrs/ADR-*.md` — 36 of 46 ADRs lack
`## Alternatives rejected` and 24 lack `## Verification`, the two sections most
correlated with decision quality. Hooks are the heaviest lift (154 files, only 15
have a `PURPOSE/DESCRIPTION` header, and 83 lack `set -euo pipefail`); a hook
contract check belongs in the already-existing `test_hooks_contracts.py` rather
than a new file. Rules are the cheapest: only 4 of 107 lack the SCOPE comment
(the highest-signal field), and a structural section check can extend the existing
`test_rules_enforcement.py`. The biggest open question for the operator: are
`## Alternatives rejected` and `## Verification` hard-required for ALL ADRs, or
only for ADRs written after a cutoff date?

## Proposed Contract — rules/*.md

| Field | Required? | Justification |
|---|---|---|
| `<!-- SCOPE: os-only\|project\|both -->` HTML comment, line 1 | **Yes** | Already present in 103/107; 4 gaps are fast backfill; drives cross-harness-authoring gate |
| H1 title (`# <Rule Name>`) | **Yes** | Present in 107/107 — cost is zero; ensures catalog parsability |
| Opening section (`## Purpose` OR `## Rule` OR `## Principle` OR `## Mandate`) | **Yes (any one)** | 54+15 = 69/107 have at least one; the other 38 files open with a prose block — allow the section to carry any of these labels |
| Status qualifier | **Maybe** | Only 1/107 has `## Status`; ADR-067 Phase 1 skipped this for skills too; add as optional metadata comment (`<!-- STATUS: always-active|contextual -->`) to avoid breaking the 107-file baseline |
| `## Contextual Trigger` (for contextual rules) | **Conditional** | Present in 47/107; only meaningful for contextual rules; make it required IF the SCOPE comment says `contextual` or the body says "Contextual Trigger" already — not a global requirement |
| Examples section | **No** | High toil; present in ~30/107; better enforced by convention than contract |
| Related / Integration section | **No** | Useful but low-signal for contract enforcement |

**Minimum contract**: SCOPE comment + H1 title + one recognized opening section.
Backfill cost: ~4 SCOPE fixes + ~38 opening-section wraps = ~1 hour of agent time.

## Proposed Contract — hooks/*.sh

| Field | Required? | Justification |
|---|---|---|
| `# SCOPE: os-only\|project\|both` | **Yes** | Present in 154/154 — already the de-facto convention; zero backfill cost; primary driver for cross-harness deploy decisions |
| `# PURPOSE: <one-line description>` | **Yes** | Present in only 15/154 (10%); hooks without it are opaque to reviewers and audit scripts; backfill cost is high (139 files) but LOW per-file (one line); recommend requiring for NEW hooks only, grandfather existing |
| `# EVENT: PreToolUse\|PostToolUse\|Stop\|…` | **Yes (new hooks)** | Present in 38/154 via `CONCERNS` or implicit event name in comment; needed for harness-agnostic routing (ADR-064); same grandfather policy |
| `# EXIT_CODES: 0=ok 2=block` | **Maybe** | Present in 0/154; directly maps to ADR-067 §Failure semantics; adds self-documentation; low per-file cost; recommend requiring for hooks that can exit non-zero |
| `set -euo pipefail` (or explicit opt-out comment) | **Yes** | Present in only 71/154 (excluding _lib); 83 hooks without it are failure-silent; highest-safety-impact contract item; grandfather policy for existing; BLOCK for new hooks |
| Idempotent/killswitch guard (`source _lib/killswitch_check.sh`) | **Conditional** | Present in ~30/154; required when hook is marked non-critical; audit can check for hooks that do NOT source killswitch yet have ADR-028 §584 citations |
| `# ASYNC: true\|false` | **No** | Present in 1/154; niche; not worth enforcing globally |

**Minimum contract (new hooks)**: SCOPE + PURPOSE + EVENT + `set -euo pipefail`.
**Minimum contract (existing hooks)**: SCOPE only (already met); PURPOSE+EVENT+pipefail are grandfather-exempt.
Backfill cost (if applied to all): 139 PURPOSE lines + 116 EVENT lines + 83 pipefail additions = 3-6 hours of agent time.

## Proposed Contract — docs/adrs/ADR-*.md

| Section | Required? | Current coverage (of 46) | Justification |
|---|---|---|---|
| `## Status` | **Yes** | 25/46 (54%) | Minimum governance signal; critical for decision tracking; low backfill cost (one line per missing ADR) |
| `## Context` | **Yes** | 34/46 (74%) | Without context, decision rationale is lost; must have ≥2 sentences |
| `## Decision` | **Yes** | 37/46 (80%) | Core artifact; must contain ≥1 table or ≥3 bullet points to avoid "Decision: we decided to do it" stubs |
| `## Consequences` | **Yes** | 32/46 (70%) | Positive AND negative sub-sections required for balance; ≥1 item each |
| `## Alternatives rejected` | **Yes (new ADRs); Maybe (backfill)** | 10/46 (22%) | Highest decision-quality signal; prevents revisiting rejected paths; operator should decide on cutoff date for backfill |
| `## Verification` | **Yes (new ADRs); Maybe (backfill)** | 22/46 (48%) | Drives Phase 1's own verification model; at least 1 runnable command required |
| `## Related` | **No (but encouraged)** | 19/46 (41%) | Useful for navigation; too noisy to enforce contractually |
| `## Open questions` | **No** | 19/46 (41%) | Valuable but inherently absent in finalized ADRs; enforce only as optional |
| Monotonically increasing filename number | **Yes** | All existing OK | `ADR-NNN` where NNN > previous max; auditable by sort |

**Minimum contract (new ADRs)**: Status + Context + Decision + Consequences + Alternatives rejected + Verification.
**Minimum contract (existing ADRs)**: Status + Context + Decision + Consequences (the 4 foundational sections).
Backfill cost at minimum contract: 21 Status + 12 Context + 9 Decision + 14 Consequences = ~56 sections across 46 ADRs = 2-4 hours.

## Recommended Phase 2 Implementation Order

1. **`docs/adrs/ADR-*.md`** — highest ROI. Creating one audit test (`test_adr_contracts.py`) + one template catches future drift on the most governance-critical artifact type. Backfill of Status+Context+Decision+Consequences on ~20 older ADRs is a contained agent task.

2. **`rules/*.md`** — second priority. Extending `test_rules_enforcement.py` with a structural section check is a 20-line code change. Backfill of 38 opening-section wraps is mechanical.

3. **`hooks/*.sh`** — last. The existing `test_hooks_contracts.py` already provides substantial coverage. The new additions (PURPOSE/EVENT for new hooks, pipefail gate) are enforced at authoring time via the template, not by mass backfill. Lowest urgency.

Rationale for ADR-first: the missing `## Alternatives rejected` and `## Verification` sections are not cosmetic — they are what enable future operators to understand WHY decisions were made and HOW to verify they hold. Rules drift is already partially caught by the existing enforcement test. Hook drift is partially self-limiting (hook dies silently vs. breaks CI). ADR drift is invisible until someone reads an old ADR months later.

## Open Questions for Operator

1. **ADR backfill cutoff**: Should `## Alternatives rejected` be required for ADRs prior to ADR-067, or only for ADR-067 and later? The 36 pre-067 ADRs span a range from thorough (ADR-064 has 6 rejected alternatives) to sparse (ADR-027 is a 1-paragraph decision). A cutoff avoids toil but leaves old decisions opaque.

2. **Hook grandfather boundary**: The grandfather policy (existing 154 hooks exempt from PURPOSE/EVENT/pipefail) is simple but creates a two-tier system. Is there a preferred cleanup sprint to backfill the 83 hooks missing `set -euo pipefail`, or should that remain informal best-effort?

3. **Strict mode (`COS_STRICT_RULE_VALIDATION=1`)**: Should the PostToolUse hooks for rules and ADRs mirror the `COS_STRICT_SKILL_VALIDATION=1` pattern from Phase 1, or start strict (exit 2) by default given that rules and ADRs are lower-velocity than skills?

4. **Template enforcement in `/add-rule` and `/add-hook` skills**: Both skills exist. Do they currently consume any template? If not, is the operator OK with modifying those skill files in Phase 2 (touching 2 SKILL.md files in `skills/`)?

5. **Monotonic ADR numbering audit**: The proposed audit checks numeric monotonicity in ADR filenames. There are already gaps in the sequence (ADR-040 appears to be missing, ADR-027a/028a/b/c are lettered variants). Should the audit BLOCK on gaps or WARN?
