# Contributing to Cognitive OS

Thank you for your interest in contributing to Cognitive OS! This guide will help you get started.

## How to Contribute

### Reporting Bugs

1. Check [existing issues](https://github.com/luum-home/luum-cognitive-os/issues) to avoid duplicates
2. Use the **Bug Report** issue template
3. Include: OS, shell, active harness (Claude Code, Codex, Cursor, etc.), and steps to reproduce
4. Attach relevant logs from `.cognitive-os/metrics/` if applicable

### Suggesting Features

1. Open a **Feature Request** issue
2. Describe the problem you're solving, not just the solution
3. Explain how it fits Cognitive OS's 3-layer architecture (core / project / generated)

### Submitting Code

1. Fork the repository
2. Create a feature branch from `main`: `git checkout -b feat/my-feature`
3. Make your changes following the guidelines below
4. Run the test suite: `bash scripts/run-all-tests.sh`
5. Commit with a clear message (see Commit Messages below)
6. Open a Pull Request using the PR template

## Development Setup

```bash
# Clone your fork
git clone https://github.com/YOUR_USERNAME/luum-cognitive-os.git
cd luum-cognitive-os

# (Optional) Start infrastructure for full integration testing
docker compose -f docker-compose.cognitive-os.yml up -d

# Run the full suite
bash scripts/run-all-tests.sh
```

## Architecture Guidelines

Cognitive OS has a strict separation between **core** and **project-specific** code:

| Directory | Scope | Rule |
|-----------|-------|------|
| `.cognitive-os/` | Universal framework | Must work for ANY project. No project-specific references. |
| `.claude/` | Project-specific | Tied to a specific project. Generated by `/cognitive-os-init`. |
| Root `rules/`, `skills/`, `hooks/` | Core source content | Product runtime content that gets projected into installed environments. |

**Before submitting, ask yourself:**
- Does my change belong in `.cognitive-os/` (core) or is it project-specific?
- Will this work for Go, Node.js, Python, Rust, and any other stack?
- Am I adding a dependency that breaks portability?

## Code Style

### Shell Scripts (hooks, tests)
- Use `#!/usr/bin/env bash` shebang
- Use `set -euo pipefail` (or `set -uo pipefail` for test runners)
- Quote all variables: `"$VAR"` not `$VAR`
- Prefer `COGNITIVE_OS_PROJECT_DIR` or canonical project-root helpers for paths
- Do not introduce new Claude-only environment assumptions outside harness adapters or compatibility code

### YAML (configs, squads, agents)
- 2-space indentation
- Comments explain "why", not "what"

### Markdown (rules, skills, docs)
- Skills follow the `SKILL.md` format with frontmatter
- Rules follow the `rules/*.md` format
- Keep lines under 120 characters

## Commit Messages

Use conventional commit format:

```
feat: add progressive skill loading
fix: circuit breaker cooldown not resetting
docs: clarify hook registration process
test: add integration test for auto-repair chain
refactor: extract shared hook library functions
```

## Testing

All contributions should include tests when applicable:

- **Unit and contract tests**: `python3 -m pytest tests/unit tests/contracts -q`
- **Behavior tests**: `python3 -m pytest tests/behavior -q`
- **Go core tests**: `go test ./internal/provider/... ./internal/validator/... ./pkg/hook/...`
- **Full suite**: `bash scripts/run-all-tests.sh`

Run the full suite before submitting:

```bash
bash scripts/run-all-tests.sh
```

## Pull Request Process

1. Ensure all tests pass
2. Update documentation if you changed behavior
3. Fill in the PR template completely
4. A maintainer will review your PR — we aim to respond within 48 hours
5. Address review feedback with new commits (don't force-push during review)

## What We're Looking For

High-impact contributions:
- New universal skills that work across any stack
- Improvements to the auto-repair (MAPE-K) system
- Better hook chain performance
- Documentation and examples
- Bug fixes with regression tests
