# ADR-118 / ADR-121 / ADR-123 Concrete Slice Backlog

> Status: execution backlog
> Updated: 2026-05-05
> Related ADRs: ADR-118, ADR-121, ADR-123

## Purpose

ADR-118, ADR-121, and ADR-123 are deferred because they are program umbrellas,
not single missing features. This backlog converts them into bounded slices that
can be implemented, reviewed, validated, and landed independently without
claiming the entire program is complete.

## Execution Rules

1. Implement one slice at a time unless two slices have disjoint files and tests.
2. Each slice must update the owning ADR or plan with implementation evidence.
3. A slice is not complete until every listed acceptance criterion has passed.
4. Slices that introduce blocking behavior must start in observe or warn unless
   the acceptance criteria include false-positive coverage for block mode.
5. Slices must not add provider-backed, Docker, VM, Kubernetes, or chaos-heavy
   work to default lanes.

## Suggested Implementation Order

1. ADR-118-S1 — scenario manifest.
2. ADR-121-S3 — guard maturity contract, because it protects later blockers.
3. ADR-123-S1 — friction telemetry, because it provides prioritization data.
4. ADR-123-S4 — unified operational status.
5. ADR-121-S2 — WIP ownership inventory.
6. ADR-118-S2 and ADR-118-S3 — highest-value swarm fixtures.
7. ADR-123-S2 and ADR-123-S3 — adaptive profiles and safe repairs.
8. ADR-121-S1 and ADR-123-S5 — protected landing plus lane recommendation.
9. ADR-121-S4 and ADR-118-S4 — lane budgets and completion reconciliation.

## Non-Goals for These Slices

- Marking ADR-118, ADR-121, or ADR-123 as fully implemented.
- Adding live provider, Docker, VM, Kubernetes, or cloud proofs to default lanes.
- Replacing existing plans; this backlog narrows them into executable slices.
