# v1.0 Release Criteria

Before tagging `v1.0`, every check below MUST be green. The canary script
(`scripts/cos-release-check.sh`) automates the core of this gate; the
remaining checks are human review items.

Run the automated gate first:

```bash
bash scripts/cos-release-check.sh
# exit 0 => all automated checks passed
```

## 1. Fresh-install parity (automated)

Verified by `scripts/cos-release-check.sh` scenarios `install_default` and
`install_full`. Each must be green.

| Check | Verification |
|---|---|
| Install completes without error (default profile) | `install_default.ok == true` |
| Install completes without error (full profile) | `install_full.ok == true` |
| `.claude/settings.json` is valid JSON | `details.settings_json == "OK"` |
| Every wired hook exists on disk | `details.hooks_missing_on_disk == 0` |
| `cos-status.sh --json` reports 0 health failures | `details.cos_status_failures == 0` |
| Full profile installs at least 10 skills | `details.skills_installed >= 10` |

## 2. Upgrade idempotence (automated)

Verified by scenario `upgrade_idempotent`.

| Check | Verification |
|---|---|
| Second `self-install.sh` run produces identical state (sha256 snapshot) | `details.idempotent == true` |
| Hook count does not regress | `post_hooks >= pre_hooks` |
| Skill count does not regress | `post_skills >= pre_skills` |

## 3. Rate-limiter under load (automated)

Verified by scenario `rate_limiter_load`. 30 invocations spawned in a tight
loop against `hooks/rate-limiter.sh`.

| Check | Verification |
|---|---|
| No invocation crashed (exit code not in `{0, 2}`) | `details.crashed == 0` |
| Rate-limiter throttles OR allows all 30 | `passed + blocked_rate_limited == total` |
| If rate-limiter not wired in current profile, scenario reports "skipped" | `ok == true`, details is a reason string |

Note: the exact number of blocked requests is phase-dependent
(reconstruction relaxes caps, production tightens them), so we do not assert
a specific block count — only that the hook does not crash.

## 4. Core skills reachability (automated)

Verified by `scripts/cos-core-skills-check.sh`. All 10 skills MUST resolve:

- `compose-prompt`
- `exhaustive-prompt`
- `agent-dashboard`
- `auto-refine`
- `verification-before-completion`
- `plan-feature`
- `session-backlog`
- `resource-governor`
- `cos-status` (script-only — `scripts/cos-status.sh` present and executable)

For the 9 SKILL.md-backed entries, the script validates:

| Check | Verification |
|---|---|
| `SKILL.md` exists under `.claude/skills/<name>/` or `skills/<name>/` | file found |
| YAML frontmatter parses (prefers PyYAML) | no exception |
| `name:` in frontmatter matches the directory name | string match |

## 5. Pytest canary suite (automated)

```bash
python3 -m pytest tests/integration/test_fresh_install_canary.py -m canary -v
```

The suite wraps the release-check script and enforces assertions per
scenario. This is what CI should run in the "release" job.

## 6. Local Homebrew install canary (opt-in automated)

The default CI/laptop lane does **not** run a real Homebrew install because it
writes to the host Homebrew prefix. The plumbing is still checked by
`tests/integration/test_fresh_install_canary.py`; the real local canary is an
explicit release-operator step:

```bash
COS_RUN_HOMEBREW_CANARY=1 scripts/cos-homebrew-local-canary --apply
```

This copies the in-repo formula into a temporary local tap, creates a
checksummed Git HEAD source tarball from the current repository, rewrites only
the temporary formula copy to that local source, and then installs with:

```bash
brew install --build-from-source <temporary-local-tap>/cognitive-os
```

Then it verifies:

| Check | Verification |
|---|---|
| Installed binary exists | `$(brew --prefix cognitive-os)/bin/cos` is executable |
| Installed binary runs | `cos version` exits 0 |
| Formula test passes | `brew test cognitive-os` exits 0 |
| Host is returned to prior state | script uninstalls `cognitive-os` unless `--keep-installed` is passed |

This proves the local formula path. It does **not** prove the external tap path.
After `luum-home/homebrew-tap` is created and a release publishes into it, the
release operator must also verify:

```bash
brew install luum-home/tap/cognitive-os
```

## 7. Documentation checks (human review)

These are NOT enforced by the script — the reviewer must confirm them:

- [ ] `CHANGELOG.md` has a `v1.0.0` heading summarizing the release.
- [ ] `README.md` install snippet uses a pinned commit or version tag, not `main`.
- [ ] `docs/04-Concepts/architecture/harness-adoption-gap/` ADRs are all in `Status: Accepted`
      or `Superseded` — no `Status: Proposed` in the v1.0 set.
- [ ] `rules/RULES-COMPACT.md` reflects the final rule surface.

## 8. Git state (human review)

- [ ] Working tree is clean (`git status` empty).
- [ ] No TODO/FIXME/XXX introduced in the v1.0 changeset relative to the
      previous tag (`git diff <prev>..HEAD -- '*.sh' '*.py' '*.go' | grep -E 'TODO|FIXME|XXX' | wc -l` is 0).
- [ ] The release branch has been rebased onto `main`.

## Exit criteria

All items in sections 1-5 must pass the automated gate:

```bash
bash scripts/cos-release-check.sh && \
bash scripts/cos-core-skills-check.sh && \
python3 -m pytest tests/integration/test_fresh_install_canary.py -m canary && \
COS_RUN_HOMEBREW_CANARY=1 scripts/cos-homebrew-local-canary --apply
```

Sections 7-8 require sign-off in the release PR description before merge.

## Rollback trigger

If any automated check starts failing after tagging, `cos-update.sh` offers
rollback. See `scripts/cos-update.sh --help` for the `--auto-rollback` flag.
