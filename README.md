Cognitive OS is the operational layer for AI coding agents. It makes agents more governable, verifiable, and portable in real repositories without forcing teams to become agent-infrastructure experts.

**Simple to start, rigorous under the hood**: the durable core is hooks, context, policy, package contracts, capability profiles, compatibility drivers, and outcome metrics. Skills, dashboards, squads, auto-repair, and other advanced systems are optional layers that must not blur the core product promise.

**Universal**: Cognitive OS works for any project - Go, Node.js, Python, Java, Rust, or any stack. Project-specific content is generated by `/cognitive-os-init` and projected through the active harness driver, not hardcoded into the OS itself.

```
$ /path/to/luum-agent-os/install.sh --harness=codex
Installing Cognitive OS into .cognitive-os/...
Cognitive OS installed!

$ COGNITIVE_OS_PROJECT_DIR="$PWD" bash /path/to/luum-agent-os/scripts/cos-status.sh
PASS    active settings driver is valid
PASS    wired hooks exist

$ claude  # or another supported harness driver
> /cognitive-os-init
Detecting stack... Node.js + Express found
Generating project-specific rules, skills, and hooks...
Done! 3 rules, 5 skills, 4 hooks generated in .claude/

Your AI now has memory, quality gates, and self-healing.
```

## Why "Cognitive OS"

Cognitive OS is not a metaphor — it's an architecturally accurate name. Like a traditional operating system manages hardware resources, Cognitive OS manages **cognition**: tokens, context, decisions, and agent coordination.

| OS Concept | Cognitive OS Equivalent |
|---|---|
| **Kernel** | `cognitive-os.yaml` + `hooks/_lib/` (core runtime) |
| **Process Scheduler** | Hook chain (SessionStart → PreToolUse → PostToolUse → Stop) |
| **Memory Management** | Engram (persistent) + context management (session) |
| **File System** | Metrics JSONL + remediation registry + transcripts |
| **Device Drivers** | Skills (interface between the agent and tools) |
| **System Calls** | Rules (contracts that processes must respect) |
| **Networking** | Harness drivers + provider adapters (tool and model portability) |
| **Self-Healing** | MAPE-K loop (like a kernel watchdog) |
| **Package Manager** | Tool discovery + skill auto-generation |
| **Init System** | `cognitive-os-init` (like systemd) |

The difference: a traditional OS manages hardware. Cognitive OS manages **cognition** — the resources are tokens, context, and decisions, not CPU and RAM. But the architecture is isomorphic: abstraction layers, lifecycle management, resource governance, fault tolerance.

## Quick Start (Any Project)

> **Important**: Always run the installer FROM your project directory.
> The installer creates `.cognitive-os/` and projects settings for the selected harness in the **current working directory**.

```bash
# Step 1: cd into YOUR PROJECT (not the Cognitive OS repo)
cd /path/to/your/project

# Step 2: Run the installer (pick a harness)

# Option A: Remote install (from GitHub)
curl -sL https://raw.githubusercontent.com/luum-home/luum-cognitive-os/main/install.sh | bash -s -- --harness=claude

# Option B: Local install (from a cloned repo)
/path/to/luum-agent-os/install.sh --harness=codex

# Option C: Force overwrite an existing installation
/path/to/luum-agent-os/install.sh --force --harness=codex

# Step 3: Inspect what is active
COGNITIVE_OS_PROJECT_DIR="$PWD" bash /path/to/luum-agent-os/scripts/cos-status.sh

# Step 4: Initialize in Claude Code when using the Claude driver
claude
> /cognitive-os-init

# (Optional) Start infrastructure for observability
docker compose -f .cognitive-os/docker-compose.cognitive-os.yml up -d
```

The installer auto-detects whether it is run from within the Cognitive OS repo (local mode) or via `curl | bash` (remote mode). Run `install.sh --help` for full usage.

> **`--from` flag**: Only needed when the `install.sh` script is not inside the Cognitive OS repo (e.g., downloaded separately). In that case, use `--from /path/to/luum-agent-os` to tell the installer where to find the source files. If you run the script directly from the repo, `--from` is unnecessary.

## Keeping Projects Updated

When you update the COS source repo, all installed projects update automatically:

```bash
# As a maintainer: push triggers auto-update of your local projects
git push

# As a user: pull triggers auto-update of your local projects
git pull
```

The auto-update is powered by git hooks (`post-merge` + `pre-push`) that re-run the installer on every registered project. Install the hooks once with `bash scripts/setup-git-hooks.sh`. See [docs/getting-started.md](docs/getting-started.md) for details.

## 3-Layer Architecture

```
Layer 1: Cognitive OS (universal)  -> .cognitive-os/        (copy to any project)
Layer 2: Project extensions       -> {project}/.claude/ (project-specific)
Layer 3: Generated from config    -> /cognitive-os-init     (auto-detected)
```

See [docs/os-vs-project-separation.md](docs/os-vs-project-separation.md) for details.

## Docker Architecture (3 Layers)

```bash
# Layer 1: Your project infrastructure
docker compose up -d

# Layer 2: Your application services
docker compose -f docker-compose.services.yml up -d

# Layer 3: Cognitive OS (plug-and-play, add to ANY project)
docker compose -f docker-compose.cognitive-os.yml up -d
```

Cognitive OS services are **completely isolated** -- they share the Docker network but have zero dependencies on your application services. Remove Cognitive OS = zero impact on your app.

## Setup Steps (Detailed)

1. Copy the entire `.cognitive-os/` directory to your project root
2. Project harness settings with `--harness=claude` or `--harness=codex`
3. Run `/cognitive-os-init` in Claude Code when using the Claude driver
4. Edit `.cognitive-os/cognitive-os.yaml` to adjust phase, quality thresholds, etc.
5. Run `COGNITIVE_OS_PROJECT_DIR="$PWD" bash /path/to/luum-agent-os/scripts/cos-status.sh` to verify active hooks, skills, and rules

## Hook Registration

Hooks in `hooks/` are registered through the active harness settings driver.
For Claude Code, `.claude/settings.json` uses:
```json
{
  "command": "bash \"$CLAUDE_PROJECT_DIR/hooks/<hook-name>.sh\""
}
```

Project-specific Claude hooks stay in `.claude/hooks/` and use:
```json
{
  "command": "bash \"$CLAUDE_PROJECT_DIR/.claude/hooks/<hook-name>.sh\""
}
```

## Separation of Concerns

| Directory | Scope | Portable? |
|-----------|-------|-----------|
| `.cognitive-os/` | Universal agent infrastructure | Yes - copy to any project |
| `.claude/` | Claude Code driver projection | Driver-specific |
| `.codex/` | Codex driver projection | Driver-specific |
| `.claude/CLAUDE.md` | Project instructions | No |
| `.claude/rules/` | Project rules (architecture, services) | No |
| `.claude/skills/` | Project skills (framework patterns, etc.) | No |
| `.claude/hooks/` | Project hooks (prod URL blocking, etc.) | No |

## Key Components

### Skills (universal)
Reusable agent capabilities: model optimizer, error analyzer, SRE agent, squad manager,
systematic debugging, TDD, verification, coverage enforcement, retrospective, etc.

### Rules (universal)
Always-active governance: fault tolerance, licensing, cost tracking, SRE protocol,
squad protocol, skill adaptation, error learning, phase-aware agents, etc.

### Hooks (universal)
Lifecycle hooks that fire on session start, tool use, compaction, and session end.
Handle error learning, metrics tracking, agent pre-launch, phase injection, auto-repair, and more.

### Auto-Repair (MAPE-K)
Closed-loop self-healing system. Monitors errors, analyzes patterns, plans fixes (registry lookup or LLM-generated),
executes in worktree isolation, verifies via build/test/lint, and registers successful repairs.
Includes circuit breaker (2 strikes), metrics auto-calibration, conversation memory, and tool discovery.

- **Tech Radar**: Automatic tool discovery with [Thoughtworks-style Tech Radar](reference/tool-watchlist.md) classification (ADOPT → TRIAL → ASSESS → HOLD)

### /cognitive-os-init (the bridge)
Detects your project stack and generates project-specific skills, rules, and hooks
in `{project}/.claude/`. Run once per project.

### Optional and Experimental Layers
Dashboards, squads, manager agents, and production control-plane ideas are useful
extension or experimental surfaces. They are intentionally not the adoption path.
The current wedge is the operating layer that makes coding agents governable,
verifiable, and portable in real repositories.

## Competitive Landscape

No single tool provides what Cognitive OS does. The closest comparison is the **stack you'd need to build yourself**:

- **Code agent** (Claude Code, Aider, Cursor) + **Self-healing** (StackStorm) + **Memory** (custom DB) + **Governance** (custom CI/CD) + **Metrics** (Grafana) + **Multi-agent** (CrewAI) = what Cognitive OS provides in one system.

See [docs/benchmarking.md](docs/benchmarking.md) for benchmark mechanics and [docs/competitive-landscape.md](docs/competitive-landscape.md) for detailed market comparisons.

## Configuration

Main config in `cognitive-os.yaml`:
- `project.phase`: Current lifecycle phase (reconstruction, stabilization, production, maintenance)
- `project.name`: Project identifier
- `project.type`: Industry type (fintech, ecommerce, saas, webapp, healthcare)
- `project.infrastructure`: Detected services, databases, auth, cache, messaging
- Squad references, model routing, quality gates
