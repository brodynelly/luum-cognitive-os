# ADR-189: Harness Implementation Coverage for Agentic Primitives

## Status

Accepted — 2026-05-06

## Context

ADR-019 and the scope-classification governance primitive classify every agentic primitive by intended audience:

```text
os-only | project | both
```

That axis is necessary but not sufficient. A primitive marked `both` can be valid for both the Cognitive OS repository and consumer projects while still not being implemented the same way in every agent IDE or harness. Claude Code and Codex are the forcing example: Claude Code exposes runtime events such as `SubagentStart`, `PreCompact`, `TeammateIdle`, and task events that Codex does not currently expose through the same hook projection surface. Conversely, Codex has a dedicated adapter hook (`concurrent-write-guard-codex-proxy.sh`).

The previous wording invited overclaiming: "scope=both" could be read as "works the same in Claude Code and Codex." That is false. Scope declares portability intent; harness coverage proves effective implementation.

## Decision

Add a second, mandatory classification axis for agentic primitive implementation:

```text
scope: os-only | project | both
×
family: hook | skill | rule | script | template
×
harness: claude | codex | shell-ci | cursor | vscode | etc.
×
state: installed | projected | wired | executable | behavior-proven
```

The canonical machine-readable report is:

```text
docs/reports/primitive-harness-coverage-latest.json
```

It is generated by:

```bash
python3 scripts/primitive_harness_coverage.py --project-dir .
```

Each row records:

- primitive path;
- primitive family;
- scope marker;
- per-harness state;
- effective coverage label;
- explicit gap when scope intent is stronger than implementation evidence.

Example row shape:

```json
{
  "primitive": "hooks/session-init.sh",
  "scope": "both",
  "family": "hooks",
  "harnesses": {
    "claude": {"installed": true, "projected": true, "wired": true, "events": ["SessionStart"]},
    "codex": {"installed": true, "projected": true, "wired": true, "events": ["SessionStart"]},
    "shell-ci": {"installed": true, "projected": false, "wired": false, "commands": []}
  },
  "coverage": "claude+codex",
  "gap": null
}
```

## State semantics

| State | Meaning |
|---|---|
| `installed` | The primitive file exists in the measured project/source tree. |
| `projected` | A harness/profile exposes the primitive structurally, such as rules/skills or shell command projection. |
| `wired` | A runtime hook is connected to a harness event. |
| `executable` | The primitive can be invoked as an executable script/hook. |
| `behavior-proven` | Automated or manual tests reference the primitive directly enough to count as behavior evidence. |

## Examples

| Primitive | Scope | Claude | Codex | Interpretation |
|---|---|---|---|---|
| `hooks/session-init.sh` | `both` | wired | wired | Portable runtime primitive. |
| `hooks/pre-compaction-flush.sh` | `both` | wired | no equivalent event | Partial portability / Claude-only behavior until Codex has an equivalent lifecycle event. |
| `hooks/concurrent-write-guard-codex-proxy.sh` | `both` | no | wired | Codex adapter primitive. |
| `rules/RULES-COMPACT.md` | `both`/`project` | installed/projected | installed/projected | Context-portable rule surface, not a runtime hook. |
| Shell-CI scripts | `both`/`project` | no hook | no hook | Command surface, not IDE event surface. |

## Consequences

- Product claims must distinguish scope intent from harness implementation.
- A `both` primitive can have a harness gap; this is visible debt, not a contradiction.
- Claude/Codex parity can be measured without pretending they expose identical events.
- Planned IDEs such as Cursor or VS Code Copilot can remain structural/context projections until native lifecycle proof exists.
- ACC and future dashboards can consume one report instead of re-deriving harness support from settings files.

## Alternatives rejected

| Alternative | Why rejected |
|---|---|
| Extend `SCOPE: both` semantics to mean every IDE behaves identically | False for current Claude/Codex event models and would make useful portable primitives look invalid. |
| Keep harness differences in prose only | Prose does not ratchet, does not feed ACC, and cannot prevent overclaiming. |
| Maintain one manually curated matrix | Too easy to drift as settings drivers and projection profiles change. |

## Acceptance Criteria

```text
ACCEPTANCE CRITERIA:
1. `python3 scripts/primitive_harness_coverage.py --project-dir .` writes JSON and Markdown reports.
2. Each row includes scope, family, harness states, coverage, and gap.
3. Unit tests prove the Claude/Codex/Shell-CI examples and drift semantics on synthetic fixtures.
4. Contract tests prove the repository report is generated and includes required state semantics.
5. Manual test documents how an operator inspects Claude-vs-Codex differences.
6. A skill exists so agents can invoke the report intentionally instead of guessing from `SCOPE`.
```
