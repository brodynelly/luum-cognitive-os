# ADR-028: SO Reliability & Observability Framework

## Status
ACCEPTED (2026-04-21) — Full 6-pillar framework CLOSED. Addenda ADR-028a/b/c resolved all PENDING items (commit 423bd86). Originally proposed 2026-04-17.

## Context

On 2026-04-17, while preparing ADR-027 (SO Slimming), three structural
failures surfaced in the SO itself. Each, taken alone, reads as a bug.
Read together, they describe the absence of an SRE layer for the agent
runtime.

**Bug 1 — resource leak (`hooks/session-init.sh:120-128`).**
The now-disabled baseline block ran `python3 -m pytest` (full suite) on
every `SessionStart` via `( … ) &`. Children inherited the parent's file
descriptors; the parent stayed blocked in `S` state. Across accumulated
sessions: **190 orphaned `session-init.sh`**, **187 orphaned `pytest`**,
and **49 orphaned `pre-commit-gate.sh` / `coverage-report.sh`** — ~350
orphans retaining ~300 MiB. Authored 2026-04-10 in commit `1b755cf`
(`feat(ws11): test baseline diff hook for anti-confirmation-bias`),
good intent, broken implementation. Disabled in this session; the block
now writes a pointer to ADR-027 Phase 3 (see `session-init.sh:120-125`).
**No hook test caught it.** Hook tests assert exit-code and stdout; none
count surviving children, FDs, or RAM delta.

**Bug 2 — reset cascade.**
`git reflog` for this session shows a series of
`HEAD@{n}: reset: moving to HEAD` entries during a window when symlink
fixes to `hooks/self-install.sh` were in flight. The working tree was
repeatedly wiped without author intent. The culprit is still unidentified
but is almost certainly a hook running `git reset --hard HEAD` as part
of a "cleanup" path. The destructive operation had no coordination with
other hooks and no audit trail beyond `reflog`.

**Bug 3 — stale git lock.**
`.git/index.lock` was observed held for ~2 h with no live git process
owning it — another orphan from a crashed hook chain. Unlike Bug 1 it
did not leak RAM; it locked VCS.

The common thread: the SO spawns, never reliably reaps, sometimes
destroys, and emits no runtime telemetry that would have made any of the
three detectable within one session. This ADR treats that as a
framework-level gap, not three isolated bugs.

Root causes:

1. **No runtime observability of the SO itself.** JSONL metrics track
   business concerns (cost, tokens, trust scores) but not process
   count, FDs, RAM per hook, nor orphan rate.
2. **No contract tests for hooks.** "Does what it says" is verified;
   "leaves the system clean" is not.
3. **Well-intentioned features added in isolation.** Baseline capture,
   `coverage-report`, `pre-commit-gate`, MCP-always-on, `self-install`
   — each has a local owner, none has a cross-hook coordinator.

Full context, phase sketches, and the 6-pillar concept this ADR formalises
live in `.cognitive-os/plans/roadmaps/so-reliability-mega-plan.md`. This ADR turns that plan
into binding decisions and does **not** duplicate its narrative.

## Scope

**In scope:**

- Hooks and any subprocess a hook spawns.
- Claude Code sub-agents (lifecycle, liveness, heartbeat).
- `.cognitive-os/metrics/*.jsonl` writers/readers/retention.
- MCP server lifecycle (startup discipline, idle monitoring).
- SLOs for the above.
- Incident runbook, emergency kill-switch, chaos suite.
- New rule `rules/hook-contracts.md` (Pillar 2).
- Observability contract every new hook must satisfy.

**Out of scope:**

- Rewriting the orchestrator. Covered by existing SDD workflow.
- MCP server internals. Their reliability is their own concern.
- Claude Code harness issues (Claude Helper RAM, OrbStack VM).
- Anything ADR-027 already owns in Phases 1/2 — this ADR amends the
  Phase 3 surface only (see *Integration with ADR-027*).

## Decisions

### D1 — Runtime Observability of the SO (Pillar 1)

**Rationale.** Bug 1 survived months because process count, RSS, FDs, and
orphans were invisible. A single CLI that prints those numbers would have
surfaced it in one session. Observability is a prerequisite for Pillars
2–6, not an add-on.

This decision is split into four sub-decisions that map 1:1 to
`SO-RELIABILITY-MEGA-PLAN.md` §"Pillar 1".

#### D1.A — Metrics census and `MetricEvent` base schema

> **Note (2026-04-18)**: Baseline claim about `hook-health.jsonl` ~40% unparseable rows refuted by metrics census — file is currently clean. See ADR-028a §5.1. The MetricEvent schema migration remains valuable; only the motivation example changed.

**Artefacts:**

- `docs/06-Daily/reports/metrics-census.md` — triage table with columns
  `file · writers · readers · retention_policy · status`. `status ∈
  {CONSUMED, ORPHAN, MISCONFIGURED}`. Source list of files enumerated in
  the mega-plan §"Sub-pillar 1.A".
- `lib/metric_event.py` — NEW. Defines a `MetricEvent` dataclass with
  fields `timestamp` (ISO-8601), `source` (hook/skill/agent id),
  `event_type`, `severity`, `payload` (dict), `schema_version`. All new
  JSONL writers MUST extend it. Existing non-uniform schemas (e.g.
  `cost-events.jsonl` has 2 incompatible shapes — see
  `docs/06-Daily/reports/metrics-census.md` F-3) migrate opportunistically.
- Rotation policy enforced by `hooks/rotate-metrics.sh` (ADR-027 Phase 3
  already owns this file; this ADR amends the thresholds): size > 1 MiB
  OR age > 7 days. Archive to `.cognitive-os/metrics/archive/`.

**Acceptance:**

- Every JSONL file listed in the census has a `status`.
- `ORPHAN` files either deleted or owner assigned within one sprint.
- All new writers pass `lib/metric_event.validate()` in their unit test.
- Rotation threshold: 1 MiB (tighter than ADR-027's initial 2 MiB; see
  *Integration with ADR-027*).

**Rollback:** `metrics.unified_schema: false` in `cognitive-os.yaml` —
writers fall back to per-hook ad-hoc schemas; validator becomes advisory.

#### D1.B — Process registry + reaper

**Contract.** Any hook that uses `&`, `nohup`, `disown`, or launches a
long-lived subprocess (`pytest`, `pip`, `docker`, `python` daemons, etc.)
MUST register its PID within 200 ms of spawn via
`lib.process_registry.register(pid, owner, ttl_seconds, kind)`.

- `kind ∈ {short_lived, detached_daemon}`. `detached_daemon` is
  whitelisted against the reaper but monitored for RAM/CPU.
- Records persisted to `.cognitive-os/runtime/processes.jsonl` (append) +
  `.cognitive-os/runtime/processes-live.json` (overwrite snapshot).

**Reaper.** `scripts/so-reaper.sh`:

- Runs at `SessionEnd` (new hook matcher) and as a user-level cron
  every 5 min.
- Policy: `SIGTERM`, 10 s grace, `SIGKILL`. Refuses to kill anything not
  in the registry (safe-kill). Orphans detected by
  `ps -o ppid,pid,command | grep -f <known_hook_basenames>` and not in
  registry → logged as `orphan_detected` MetricEvents, NOT auto-killed
  in Phase A; auto-kill gated behind `runtime.reaper.autokill_orphans:
  false` default.

**Artefacts:**

- `lib/process_registry.py` — NEW.
- `scripts/so-reaper.sh` — NEW.
- `hooks/session-end-reap.sh` — NEW (`SessionEnd` matcher).
- `tests/contracts/test_process_registry.py` — NEW.

**Acceptance:**

- Given a hook that `( sleep 60 ) &` without registering, the reaper
  tests MUST fail.
- Given registered PIDs with `ttl=30s`, reaper SIGTERMs after TTL and
  SIGKILLs after 10 s grace in a sandbox.
- Zero SIGKILLs to processes NOT in the registry during 1 week of
  production use (SLO; breach → incident).

**Rollback:** `runtime.reaper.enabled: false` — registry keeps logging
but no processes are killed.

#### D1.C — Sub-agent liveness (heartbeat protocol)

> **Scope note (coordination with WS13, per ADR-028a §2):** WS13
> (`lib/state_heartbeat.py`, `hooks/state-heartbeat.sh`, committed `65e4d0c`)
> provides session-level state persistence every 10 tool calls to
> `.cognitive-os/sessions/{id}/state-snapshot.json`. This is NOT a substitute
> for the D1.C agent heartbeat. WS13 answers "what was the session doing?"
> (crash recovery); D1.C answers "is this agent alive right now?" (liveness
> watchdog). The two mechanisms MUST coexist. `agent-watchdog.sh` (Phase A
> artefact) MUST NOT read `state-snapshot.json`; it reads only
> `.cognitive-os/tasks/*.heartbeat`. `crash-recovery.sh` MUST NOT read
> `.heartbeat` files as its primary source; it reads only
> `state-snapshot.json`.

Every sub-agent launched by the orchestrator writes a heartbeat every
60 s for its entire lifetime.

**Heartbeat protocol spec:**

```
Path:    .cognitive-os/tasks/{agent_id}.heartbeat
Format:  single-line JSON, overwrite each beat
Schema:  {
  "agent_id":   string,   # UUID from orchestrator
  "parent_sid": string,   # session id that launched it
  "model":      string,
  "pid":        int,
  "started_at": iso8601,
  "last_beat":  iso8601,  # updated on each write
  "state":      "launched" | "running" | "stale" | "completed",
  "phase":      string,   # free-text, current PROGRESS marker
  "beat_seq":   int       # monotonic counter
}
Interval: 60 s ± 5 s jitter. First beat within 3 s of launch.
Staleness: last_beat > 5 min old → watchdog marks state=stale
           and emits MetricEvent(event_type=agent_stale).
Reap:     state=stale + orchestrator confirms no outstanding tool
          call → SIGTERM. Grace 30 s. Then SIGKILL.
```

State machine: `launched → running → stale → reaped | completed`.

**Artefacts:**

- `hooks/_lib/agent-preamble.md` — AMEND to include the heartbeat
  protocol and a one-line cron-equivalent the agent emits itself
  (`hooks/_lib/heartbeat.sh` helper).
- `lib/agent_heartbeat.py` — NEW (writer + watchdog).
- `scripts/so-agent-status.sh` — NEW. Prints fleet: id, model, elapsed,
  heartbeat age, state.
- `tests/contracts/test_agent_heartbeat.py` — NEW.

**Acceptance:**

- Agent launched in sandbox writes first heartbeat within 3 s.
- Heartbeat age > 5 min triggers `state=stale` within 60 s.
- Agent completion (normal exit) triggers `state=completed` and removes
  the heartbeat file within 10 s.

**Rollback:** `runtime.heartbeat.enabled: false` — agents launch as
today; watchdog becomes no-op.

#### D1.D — Unified dashboard (`scripts/so-vitals.sh`)

One-shot CLI printing:

- Agents in flight (from D1.C).
- Registered short-lived processes + TTL (from D1.B).
- Orphan detection (processes matching hook basenames but unregistered).
- MCP RSS summary (reuses ADR-027 baseline scraping logic).
- JSONL sizes + rotation status (from D1.A).
- Hook runtime p50/p95 over last 100 invocations from `hook-health.jsonl`.

Exit code non-zero if any SLO in §"SLO catalogue" breached. Runnable
manually, from `/so-status` skill, and (silently, only emitting on
breach) at `SessionStart` behind flag `runtime.vitals.on_session_start`.

**Acceptance:**

- `scripts/so-vitals.sh --json` emits a `MetricEvent`-shaped object.
- When SLO breached, exit code = 1 and breach written to
  `.cognitive-os/metrics/so-vitals.jsonl`.

**Rollback:** remove the `SessionStart` registration; CLI remains but
is not called automatically.

### D2 — Contract Tests for Hooks (Pillar 2)

**Rationale.** Bug 1 was invisible to the existing test suite because
tests only verified behaviour (exit code, output). A contract test
framework that asserts runtime hygiene (orphans, FDs, RAM, p95) is the
regression net that would have failed immediately on commit `1b755cf`.

**New rule:** `rules/hook-contracts.md`. Five mandatory assertions for
every hook test:

1. Exit 0 on happy path.
2. **Zero child processes alive 100 ms after the hook returns**
   (exceptions require explicit `detached_daemon` annotation with a
   tracked PID via `lib.process_registry`).
3. **Zero FDs opened by the hook and left open**, excluding stdout/
   stderr/stdin. Measured via `lsof -p $HOOK_PID` snapshot diff.
4. RAM delta below threshold: ≤ 10 MiB for short hooks, ≤ 50 MiB for
   `SessionStart` hooks.
5. p95 duration under SLO: `SessionStart` < 2 s, `PreToolUse` < 200 ms,
   `PostToolUse` < 500 ms.

**Artefacts:**

- `rules/hook-contracts.md` — NEW, index-referenced from RULES-COMPACT.
- `tests/contracts/__init__.py`, `tests/contracts/conftest.py` — NEW.
- `tests/contracts/test_hook_no_orphans.py` — parametrised over every
  hook file in `hooks/*.sh`.
- `tests/contracts/test_hook_fd_clean.py` — `lsof` diff.
- `tests/contracts/test_hook_ram_delta.py` — RSS diff.
- `tests/contracts/test_hook_runtime_slo.py` — SLO p95 gate from
  `hook-health.jsonl`.
- `tests/contracts/test_hook_no_destructive_git.py` — NEW, in response
  to Bug 2: grep each hook for `git reset --hard`, `git clean -fdx`,
  `rm -rf`; hooks that contain these must carry an explicit
  `# SAFE: ACKNOWLEDGED DESTRUCTIVE` annotation at the top.

**Acceptance:**

- Running `pytest tests/contracts/` against `main` today MUST fail (the
  historical `session-init.sh:120-128` version, cherry-picked as a
  fixture, must trigger `test_hook_no_orphans`).
- CI registers `tests/contracts/` as a blocking job.
- 100 % of registered hooks have at least one contract test row (even
  if SKIPPED with a tracked reason).

**Rollback:** `hooks.contract_tests_required: false` — CI job becomes
warn-only.

### D3 — Systematic Audit (Pillar 3)

**Rationale.** Before fixing, inventory. One opus agent, read-only, walks
all hooks, flags anti-patterns with evidence. No fixes in this decision.

**Scope.** 110 hook files (count to be confirmed at audit time via
`ls hooks/*.sh | wc -l`; the mega-plan cites 110).

**Deliverable.** `docs/06-Daily/reports/hook-audit-2026-04.md` with one row per
finding:

| field | type | example |
|---|---|---|
| `hook` | string | `hooks/session-init.sh` |
| `line_range` | string | `120-128` |
| `anti_pattern` | enum | `bg_without_pid_track` |
| `severity` | enum | `BLOCKER` / `CONCERN` / `SUGGESTION` |
| `evidence` | quoted snippet | `"( python3 -m pytest ) &"` |
| `proposed_fix_class` | enum | `register_pid_and_timeout` |

**Anti-pattern taxonomy (closed set for Phase C):**

- `bg_without_pid_track` — `&` / `nohup` without `echo $! >>`.
- `subproc_without_timeout` — external command without `timeout Ns`.
- `test_run_inside_hook` — `pytest`/`go test`/`yarn test` (moves to
  ADR-027 Phase 1 `global-verify.sh`).
- `unbounded_loop` — `while true`/`while :` without break/timeout.
- `infinite_sleep` — `sleep` with argument > 600 s or unbounded.
- `destructive_git` — `git reset --hard`, `git clean -fdx`, `rm -rf`
  of paths outside `${SESSION_DIR}` or `/tmp`.
- `unrotated_write` — appends to JSONL with no rotation owner.
- `global_state_write` — writes to `~/.*` or `/etc/*`.

**Severity tiers (adversarial-review compliant):**

- BLOCKER — produces orphans, corrupts VCS, or blocks further sessions.
- CONCERN — resource leak under load or race condition.
- SUGGESTION — style, readability, duplication with another hook.

**Acceptance:**

- Every hook file appears at least once in the report, even if the only
  row is "no findings, scanned $(date)".
- At least one finding per 20 hooks on average (adversarial-review mandate:
  zero-findings = review failed; see `rules/adversarial-review.md`).
- Report is read-only; zero hooks modified in this phase.

**Rollback:** not applicable — audit is read-only.

### D4 — Systematic Fix (Pillar 4)

**Rationale.** Findings without fixes rot. Multiple sonnet agents run
in parallel, each owning a disjoint set of hooks from D3's report.
Every fix follows a canonical pattern — no bespoke "clever" rewrites.

**Canonical patterns:**

| Anti-pattern | Canonical fix |
|---|---|
| `bg_without_pid_track` | `nohup CMD &> "$SESSION_DIR/bg.log" & PID=$!; lib_process_registry_register "$PID" "$HOOK_NAME" 60 short_lived` |
| `subproc_without_timeout` | `timeout 30 CMD \|\| { echo "timeout"; exit 0; }` |
| `test_run_inside_hook` | Remove. Defer to ADR-027 Phase 1 `global-verify.sh`. |
| `unbounded_loop` / `infinite_sleep` | Max iterations guard + `timeout`. |
| `destructive_git` | Replace with explicit no-op + `#SAFE: ACKNOWLEDGED DESTRUCTIVE` annotation only if audited. Default: BLOCK commit via pre-commit-gate. |
| `unrotated_write` | Migrate writer to `lib.metric_event` + declare rotation in `hooks/rotate-metrics.sh`. |
| `global_state_write` | Move to `.cognitive-os/runtime/` under project root. |

**Parallelism guidelines:**

- 3–5 sonnet agents, disjoint hook sets, single lock owner = this ADR's
  coordinator.
- Each agent's prompt MUST include the ADR-028 acceptance criteria and
  the canonical pattern table above.
- After each batch, contract tests (D2) MUST pass on the touched hooks
  before the next batch launches.
- No agent may touch `.claude/settings.json` — the registration surface
  belongs to ADR-027 Phase 3 (see *Integration with ADR-027*).

**Acceptance:**

- 100 % of BLOCKER findings from D3 resolved.
- ≥ 80 % of CONCERN findings resolved.
- Contract test suite passes on `main` after D4 completes.

**Rollback:** per-hook — every fix is a single commit, reversible.

### D5 — SLOs, Error Budget, Incident Runbook (Pillar 5)

**Rationale.** Without published SLOs the system cannot be said to be
"healthy" or "degraded" — only "running" or "crashed". Budgets make
trade-offs explicit.

**Artefacts:**

- `rules/so-slo.md` — NEW, contains the SLO table in §"SLO catalogue"
  below and the error budget.
- `docs/05-Methodology/runbooks/so-incident-runbook.md` — NEW. Sections: symptoms
  (slow session, high CPU, high RAM), diagnosis (`scripts/so-vitals.sh`),
  triage decision tree, kill-switch, postmortem template.
- `scripts/so-emergency-stop.sh` — NEW. Disables all non-essential
  hooks (writes to `.cognitive-os/runtime/hook-killswitch.flag`,
  respected by `hooks/_lib/killswitch_check.sh` sourced at the top of
  every hook), kills registry-tracked orphans, leaves SO in a minimal
  mode equivalent to ADR-027's `lean` efficiency profile.

**Acceptance:**

- Invoking `scripts/so-emergency-stop.sh` from a chaos test leaves the
  next `SessionStart` functional in ≤ 3 s with only the whitelisted
  critical hooks firing (credential-guard, license-guard, pre-compaction-flush).
- Runbook tested end-to-end in D6 chaos suite.

**Rollback:** `runtime.killswitch_respected: false` — hooks ignore the
flag; equivalent to deleting the flag file.

### D6 — Chaos Engineering (Pillar 6)

**Rationale.** SLOs that aren't exercised are aspirational. Lightweight
chaos tests prove the system degrades gracefully, not catastrophically.

**Suite (`tests/chaos/`):**

- `test_kill_mcp_mid_session.py` — SIGKILL `engram`; orchestrator must
  continue (degraded: mem_search returns empty, session finishes).
- `test_hook_timeout.py` — inject `sleep 60` into a copy of a hook in
  a sandbox; verify the reaper and the hook-level `timeout` kill it.
- `test_disk_full_metrics.py` — fill a tmpfs to 100 %; rotator must
  either rotate or apply backpressure without crashing the session.
- `test_fd_exhaustion.py` — open 1 000 FDs before `SessionStart`; hooks
  must degrade (skip non-essential work) rather than crash.
- `test_reset_cascade_detector.py` — simulate Bug 2: run a hook that
  issues `git reset --hard HEAD`; `test_hook_no_destructive_git` from
  D2 MUST flag it. If it doesn't, the contract test itself is broken.

**Cadence:** scheduled weekly via `mcp__scheduled-tasks` OR manually
before any release-tagged commit. Integrated into CI as an allow-fail
job first, promoted to blocking once the suite stabilises (target:
2 weeks after D6 lands).

**Acceptance:**

- All five tests green on a clean `main`.
- Each test has a measurable pass/fail — no "looks fine" assertions.

**Rollback:** `chaos.schedule_enabled: false` — weekly schedule disabled;
suite remains runnable on demand.

## Integration with ADR-027

ADR-027 defined *slimming* targets (tests, context, resources). ADR-028
defines the *framework* that makes slimming safe. Relationship:

- **ADR-027 Phase 1 (tests)** — independent. Lands before or alongside
  ADR-028 Phase A. `hooks/global-verify.sh` (ADR-027) is the canonical
  landing site for `test_run_inside_hook` fixes (ADR-028 D4).

## Observability contract for every NEW hook (style-guide fragment)

Verbatim, to be copied into `rules/hook-contracts.md`. Quoted here so
ADR-028 readers do not need to chase references.

```bash
#!/usr/bin/env bash
# SCOPE: <hook scope>
# OBSERVABILITY: ACK  (mandatory: ACK | EXEMPT:<reason>)
# DESTRUCTIVE: NO     (mandatory: NO | YES — YES requires rationale below)
set -euo pipefail

HOOK_NAME="$(basename "$0")"
HOOK_START_NS=$(date +%s%N)

# 1. Killswitch check (D5) — every hook MUST source this.
. "$(dirname "$0")/_lib/killswitch_check.sh"

# 2. Any background work MUST register its PID (D1.B).
#    Example:
#    ( some_long_command ) &
#    BG_PID=$!
#    . "$(dirname "$0")/_lib/process_registry.sh"
#    lib_process_registry_register "$BG_PID" "$HOOK_NAME" 60 short_lived

# 3. Any external subprocess MUST be wrapped in `timeout`.
#    timeout 30 some_tool ... || { echo "timeout"; exit 0; }

# 4. Any JSONL write MUST go through lib/metric_event.py (D1.A)
#    and declare its rotation owner in hooks/rotate-metrics.sh.

# 5. On exit, emit runtime telemetry (D1.D input).
trap 'lib_hook_health_emit "$HOOK_NAME" "$HOOK_START_NS" "$?"' EXIT
```

**Author checklist (must tick all before merging a new hook):**

- [ ] `OBSERVABILITY: ACK` header present.
- [ ] `DESTRUCTIVE:` header present and honest.
- [ ] Killswitch sourced as first real line.
- [ ] All `&` / `nohup` / `disown` register a PID.
- [ ] All external calls wrapped in `timeout`.
- [ ] All JSONL writes use `lib.metric_event`.
- [ ] A `tests/contracts/test_<hook>.py` exists, inheriting the five
      standard assertions from `conftest.py`.
- [ ] Hook appears in `docs/06-Daily/reports/metrics-census.md` or `hook-audit`
      report as appropriate.

## Non-goals

- Rewriting Engram, MCP servers, or the Claude Code harness.
- Auto-fixing hooks during the audit (D3 is read-only).
- Replacing MLflow / Langfuse — `so-vitals` is a runtime-hygiene channel,
  not an LLM-trace channel.
- Building a UI. Every artefact is a CLI or a file.
- Enforcing a heartbeat protocol on tools outside the orchestrator (e.g.
  `gh`, `docker`).
- Back-filling historical JSONL to the `MetricEvent` schema. Opportunistic
  migration only.
- Resolving the Bug 2 `git reset` cascade in this ADR. Tracked as an
  open question and a BLOCKER prerequisite for D3.

## Open questions

Honest list. None of these block this ADR being accepted as PROPOSED;
each blocks a specific downstream phase.

1. **Bug 2 root cause.** Which hook emits `git reset --hard HEAD`? Until
   identified, D3 cannot safely run against a dirty working tree. A
   forensic micro-task must precede D3 (grep every hook + add a hook
   pre-commit wrapper that logs and refuses destructive git ops). See R3.
2. **Heartbeat overhead.** The 60 s interval and overwrite strategy is a
   guess. Phase A must measure actual overhead (write syscalls per minute,
   IO cost) and tune before Phase D relies on it.
3. **Audit finding volume.** Whether 110 hooks produce 10 findings or
   400 is unknown. D4 batching (3–5 agents) may need to be 10 agents or
   a single agent depending on the real distribution.
4. **`MetricEvent` schema versioning strategy.** Linear integer versioning
   is assumed; a breaking change in Phase D would force a migration plan
   not yet written.
5. **Emergency kill-switch semantics under an already-broken SO.** If the
   killswitch flag itself sits on a full disk, `scripts/so-emergency-stop.sh`
   might not write it. D5 must define a fallback (environment variable
   `SO_KILLSWITCH=1` overrides the flag file).
6. **Chaos test idempotency on the developer's actual machine.**
   `test_fd_exhaustion` opening 1 000 FDs could collide with OS-wide
   limits on macOS. Needs per-host sandbox or Docker-based isolation.
7. **Integration order with in-flight ADR-027 work.** If ADR-027 Phase 3
   starts before ADR-028 Phase B lands, Phase 3's hook merges will be
   regressing into a system that lacks contract tests. The coordinator
   must gate Phase 3 on ADR-028 Phases A + B explicitly.
8. **Heartbeat for the orchestrator itself.** Sub-agents heartbeat. The
   orchestrator does not. Whether the orchestrator should emit its own
   heartbeat — and where — is deferred.
9. **WS11 anti-confirmation-bias replacement.** The `test_run_inside_hook`
   fix for `session-init.sh:120-128` disables WS11 (commit `1b755cf`).
   Before closing Phase D, confirm that ADR-027 Phase 1's `global-verify.sh`
   is configured to capture a test baseline at session start and block
   "tests pass" claims without a before/after diff. If ADR-027 Phase 1 has
   not yet landed, add an interim advisory in `hooks/session-init.sh`
   pointing to this requirement. Self-certification without diff = anti-pattern.
   After WS11 disable lands, random sampling of 5 agent completions that
   include "tests pass" claims must show every claim accompanied by
   `global-verify.sh` diff output, not a bare assertion. If any sampled
   completion lacks the diff output, ADR-027 Phase 1 integration is
   incomplete. See ADR-028a §1.
