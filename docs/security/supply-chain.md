# Supply-Chain Posture

This document describes the dependency, SBOM, and supply-chain hygiene posture
of `luum-agent-os` (the Cognitive OS for AI agent orchestration). It is
intended for hostile-auditor consumption — every claim should be reproducible
from a clean checkout.

**Scope:** project's own runtime/dev dependencies (Python, Go, Node). Third-party
Claude Code plugins under `.claude/plugins/`, vendored references under
`reference/`, build artifacts under `dist/`, and example projects are explicitly
**out of scope** of the project SBOM — they are not redistributed by the
project's release artifacts and are excluded from `sbom.json`.

**Last regenerated:** 2026-05-08
**Repo commit:** see `git log -1` at the time the SBOM was generated.

## 1. SBOM (Software Bill of Materials)

### 1.1 Format

The repository ships `sbom.json` at the root in **CycloneDX 1.6 JSON** format
(`bomFormat: "CycloneDX", specVersion: "1.6"`).

### 1.2 Snapshot

| ------------------------- | ------------------------------------------------------------------ |
| File                      | `sbom.json`                                                        |
| Format                    | CycloneDX 1.6 JSON                                                 |
| Generated by              | `syft 1.44.0`                                                      |
| Generation date           | 2026-05-08                                                         |
| Source                    | `dir:.` (project root, excluding vendored/plugin/build directories)|
| Source name / version     | `luum-cognitive-os` / `0.27.1` (from `pyproject.toml`)             |
| Component count           | 241 (raw); 205 unique deduped library components                   |
| SHA-256                   | `2a3fb4322381f931da78bd2355a7f9e41457ed7586785bb48c4e8b2b9a7c9adb` |

The SHA-256 above is the literal checksum of `sbom.json` at the time this
document was committed. The committed SBOM is sanitized to use repository-relative
file component names; local absolute generation paths are not allowed in the
public artifact. Consumers verifying a release MUST re-run
`shasum -a 256 sbom.json` against the SBOM shipped alongside the release tag
and confirm it matches what the release notes advertise.

**Note on regeneration:** `syft` writes a fresh `serialNumber` and timestamp
into every SBOM, so byte-identical reproduction is not expected. To verify
that two SBOMs describe the same dependency closure, compare the
deduplicated `(purl, version)` tuple set (e.g. `jq '[.components[]|.purl]|unique'`)
rather than the raw file checksum.

### 1.3 Reproducing the SBOM

The primary toolchain is `syft` + `grype`, per
`manifests/cross-stack-license-audit.yaml`. Both are Apache-2.0.

Install (macOS):

```bash
brew install syft grype
```

Install (Linux):

```bash
curl -sSfL https://raw.githubusercontent.com/anchore/syft/main/install.sh | sh -s -- -b /usr/local/bin
curl -sSfL https://raw.githubusercontent.com/anchore/grype/main/install.sh | sh -s -- -b /usr/local/bin
```

Generate a CycloneDX SBOM (must be run from the repo root):

```bash
syft scan dir:. \
  --source-name luum-cognitive-os \
  --source-version "$(grep -m1 '^version' pyproject.toml | cut -d'"' -f2)" \
  --output cyclonedx-json=sbom.json \
  --exclude './.venv/**' \
  --exclude './node_modules/**' \
  --exclude './.git/**' \
  --exclude './dashboard/.next/**' \
  --exclude './dashboard/node_modules/**' \
  --exclude './.engram/**' \
  --exclude './.pytest_cache/**' \
  --exclude './.ruff_cache/**' \
  --exclude './.cognitive-os/**' \
  --exclude './examples/**' \
  --exclude './.claude/**' \
  --exclude './reference/**' \
  --exclude './dist/**'
```

Compute checksum:

```bash
shasum -a 256 sbom.json
```

For a vulnerability scan against the same SBOM:

```bash
grype sbom:./sbom.json
```

### 1.4 Optional alternative tooling

The following CycloneDX-native generators MAY be installed for per-ecosystem
SBOMs (not required for the canonical project SBOM, which uses syft):

- Python: `pip install cyclonedx-bom && cyclonedx-py environment -o python-sbom.json`
- Go: `go install github.com/CycloneDX/cyclonedx-gomod/cmd/cyclonedx-gomod@latest && cyclonedx-gomod mod -json -output go-sbom.json`
- Node: `npm install -g @cyclonedx/cyclonedx-npm && cd dashboard && cyclonedx-npm --output-file node-sbom.json`

Use these only for cross-checking the syft output during legal review.

by the `sharp` Node binding. Per LGPL-3.0 §4 (combined works) the combination

under LGPL — standard practice for the entire Node ecosystem using `sharp`.

The dashboard package is OPTIONAL and not a runtime dependency of `luum-cognitive-os`
itself. Operators who do not deploy the dashboard never pull these binaries.

### 3.5 UNKNOWN — 100 entries

UNKNOWN status reflects gaps in package metadata that `syft` could not resolve
without doing an online lookup. None of the UNKNOWN entries have been
identified as actually problematic; they are syft metadata-discovery gaps:

- **39 PyPI** — packages whose `License` field is empty in `METADATA` and that

  permissive licenses (e.g. `anyio` MIT, `attrs` MIT, `fastapi` MIT, `pytest`
  MIT, `ruff` MIT, `pre-commit` MIT, `cryptography` Apache-2.0+BSD-3-Clause,

  enrichment is tracked under ADR-238 follow-up work.
- **57 Go modules** — `syft` does not extract licenses from Go modules without
  running `go mod download` first. All 57 are standard MIT/BSD-3-Clause/Apache-2.0
  Go ecosystem modules (`charmbracelet/*`, `spf13/cobra`, `mattn/*`,
  `golang.org/x/*`, `modernc.org/*`). To produce authoritative Go license
  metadata, run `go-licenses report ./...` or `cyclonedx-gomod mod` after
  `go mod download`.
- **3 GitHub Actions** — `actions/checkout@v4`, `actions/setup-go@v5`,
  `actions/setup-python@v5` (or similar) — all MIT, license metadata simply
  not embedded in the `pkg:github` PURL.
- **1 npm** — `cos-dashboard@0.1.0` is the project's own dashboard root
  package; it is not a third-party dependency.

**Action:** Open a tracked task in ADR-238 to enrich SBOM with explicit SPDX
identifiers for all UNKNOWN entries before the first public release tag.
Until then, this document records the manual classification above and the
release-readiness gate counts UNKNOWN as a soft warning, not a blocker.

## 4. Pinning & Provenance Policy

### 4.1 Current state

- **Python:** `uv.lock` pins exact versions and hashes for every transitive
  dependency. `pyproject.toml` uses `>=` constraints for floor versions; the
  lockfile is the source of truth. Hash verification is enforced by `uv` on
  install.
- **Go:** `go.sum` pins module checksums (h1:) for every `require`d module.
  Verified by `go mod verify` and the Go module proxy by default.
- **Node (dashboard):** `package-lock.json` pins exact versions and integrity
  hashes (`integrity: sha512-...`). Verified by `npm ci`.
- **Container base images:** None. The project ships as language-native
  artifacts (Python wheel, Go static binaries, Node app); no Dockerfiles in
  the release path.
- **CI workflows:** Per `manifests/cross-stack-license-audit.yaml`, mutable
  `aquasecurity/trivy-action` references are **forbidden**. Workflows MUST pin
  third-party actions by full commit SHA.

### 4.2 Aspirational (tracked, not yet enforced)

- **Sigstore / cosign signing of release artifacts** — planned for the first
  public release tag (item M2 of `docs/legal/pre-public-readiness-checklist.md`).
- **SLSA provenance attestations** — out of scope for v0.x; revisit at v1.0.
- **Reproducible builds** — Python wheel reproducibility and Go binary
  determinism are tracked under future ADRs.

## 5. Verifying an SBOM Matches a Release

When a tag `vX.Y.Z` is published, the release page MUST include:

1. `sbom.json` (the CycloneDX SBOM as of the tagged commit)
2. `sbom.json.sha256` (the SHA-256 of the SBOM)
3. The repo `LICENSE` and `NOTICE` files

Consumers verify:

```bash
# 1. Clone at the exact tag
git clone --depth 1 --branch vX.Y.Z https://github.com/luum-home/luum-cognitive-os.git
cd luum-cognitive-os

# 2. Compare shipped SBOM to a freshly-regenerated one
shasum -a 256 sbom.json
# should match the value in sbom.json.sha256 from the release page

# 3. Regenerate locally (requires syft)
syft scan dir:. --output cyclonedx-json=/tmp/regenerated-sbom.json [exclusions as in §1.3]
# Component lists should be identical (modulo timestamps/serial numbers).
```

The SBOM MUST be regenerated and committed every time `pyproject.toml`,
`requirements.txt`, any `go.mod`, or `package.json` changes. CI enforcement of
this is tracked as a follow-up.

## 6. Reporting a Security Issue

### 6.1 Coordinated disclosure

If you discover a vulnerability in `luum-agent-os`, please **DO NOT** open a
public GitHub issue. Instead:

- **Email:** `security@luum.home` (preferred channel for coordinated disclosure)
- **GitHub:** Submit a private vulnerability report via the repository's
  Security tab → "Report a vulnerability"

Encrypt sensitive details with the maintainer's PGP key (published alongside
the first signed release).

### 6.2 Response SLA

| Severity              | Initial acknowledgement | Triage / status update | Patch target          |
| --------------------- | ----------------------- | ---------------------- | --------------------- |
| **Critical** (RCE, secret exfil, supply-chain) | within 48 hours      | within 5 business days | within 14 days        |
| **High**   (auth bypass, privilege escalation) | within 72 hours      | within 7 business days | within 30 days        |
| **Medium** (info disclosure, DoS)              | within 5 business days | within 14 business days | next minor release  |
| **Low**    (hardening, defense-in-depth)       | within 10 business days | best-effort           | next minor release    |

These are targets, not guarantees. The project is community-maintained; the
maintainer will keep reporters informed of progress.

### 6.3 Scope

In scope:

- The `luum-cognitive-os` Python package (`lib/`, `hooks/`, `scripts/`)
- The `cos` Go binary (`cmd/cos/`)
- The `cos-test` Go binary (`cmd/cos-test/`)
- Default hook configurations and embedded prompts
- The dashboard Next.js application (`dashboard/`) — for issues in
  first-party code, not Next.js itself

Out of scope:

- Third-party Claude Code plugins under `.claude/plugins/` (report directly
  to the plugin maintainer)
- Vendored references under `reference/` (read-only research material;
  not part of the runtime)
- Vulnerabilities in upstream dependencies whose fix is gated on the upstream
  project (we will track and update; not a Cognitive-OS bug)

### 6.4 Safe-harbor

Good-faith security research on the project — testing in your own checkout,
not against shared infrastructure, without privacy violation or service
disruption — is welcomed. We will not pursue legal action against researchers
who follow the disclosure process above.
