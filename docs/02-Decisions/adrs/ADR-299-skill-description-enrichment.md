# ADR-299: LLM-Driven Multilingual Enrichment of Skill Routing Intents

## Status

Accepted — 2026-05-13. Implemented in the same change.

## Context

ADR-296 made the COS skill router language-agnostic by replacing Jaccard with
a multilingual FastEmbed matcher (paraphrase-multilingual-MiniLM-L12-v2). It
indexed each skill's `description + summary_line + routing_intents`. Today
only ~19 of ~385 skills declare `routing_intents`; the remainder participate
on `description` alone.

The SkillRouter paper (arXiv 2603.22455) reports a **31–44 point accuracy
drop** when a router operates on description-only corpora vs. corpora
enriched with paraphrased user utterances. Empirically the drop is
asymmetric across languages — descriptions are almost always English, so
Spanish/Portuguese/German prompts pay the steepest tax even with a
multilingual encoder.

ADR-297 (LLM-dispatched fallback) and ADR-298 (benchmark harness) gave us
the dispatch plumbing and the measurement surface but did not fix the
underlying corpus quality. This ADR closes that gap.

## Decision

Build an offline enricher that, per skill, generates ~2 short natural-
language utterances per target language (default: en/es/pt/de/fr/it) via
`lib.dispatch.dispatch(...)` and merges them into SKILL.md frontmatter as
`routing_intents` entries tagged `auto_generated: true`.

### Contract

- **Every skill SHOULD have ≥ 5 routing_intents covering EN/ES/PT/DE/FR/IT**,
  auto-generated by this tool and manually refinable. Default
  `--intents-per-lang 2` × 6 languages = 12 entries per skill, well above
  the floor.
- **Auto-generated tagging is load-bearing.** Idempotency relies on
  `auto_generated: true` to distinguish enricher output from human-curated
  intents. Reruns without `--force` skip already-enriched skills.
- **Human curation always wins.** Existing intents WITHOUT the marker are
  never modified; new auto-generated intents are appended after them.
- **Vendor-neutral dispatch.** All LLM calls go through `lib/dispatch.py`
  (ADR-049 — Qwen primary, Claude fallback). No Anthropic-specific code path.
- **Strict-JSON contract.** The LLM is prompted for a fixed JSON shape
  (`{en: [...], es: [...], ...}`). Prose or malformed responses are logged
  to the audit trail and the skill file is NOT touched.

### Cost + safety guards

- `--cost-cap-usd` (default `$5`): cumulative cost ceiling; the batch halts
  gracefully on breach and saves partial progress. A warning is logged at
  80% of cap. 385 skills × 6 languages × ~$0.001/call ≈ $2.30 budget —
  the default leaves comfortable headroom.
- `--rate-limit-per-minute` (default `60`): wall-clock throttle between
  dispatch calls.
- `--dry-run`: prints proposed enrichments; writes nothing (verified by
  file-mtime assertions in the unit tests).
- `COS_DISABLE_ENRICHMENT=1`: kill switch — the public entry point returns
  an empty report.
- **Audit trail**: every dispatch call appends one JSONL record to
  `.cognitive-os/metrics/skill-enrichment.jsonl` (skill, languages,
  success, provider, cost_usd, latency_ms, error).

### Operator workflow

```bash
# Dry-run a single skill — see proposals only
scripts/cos-skill-description-enrich --dry-run --skills product-answer

# Enrich a small batch with a tight cap
scripts/cos-skill-description-enrich --skills code-review,plan-feature \
  --languages en,es --cost-cap-usd 0.50

# Full catalog enrich (one-off, expected ~$2 over ~10 minutes at 60 rpm)
scripts/cos-skill-description-enrich --skills all

# Re-roll auto-generated intents for a skill (preserves human-curated ones)
scripts/cos-skill-description-enrich --force --skills perf-profile
```

To revert: `git checkout -- skills/<name>/SKILL.md` (or revert the commit
that landed enrichment). Because auto-generated entries are tagged, an
operator can also delete them by hand without touching curated intents.

## Alternatives rejected

- **Leave the behavior as implicit agent instruction only.** Rejected because this ADR records a runtime/authoring contract that needs durable tests or audits rather than conversation-only memory.

## Verification

### Architectural test (deterministic, runs in CI)

`tests/unit/test_skill_description_enricher.py::test_enriched_corpus_improves_baseline_routing`
builds two `SemanticSkillMatcher` instances over a 5-skill held-out subset
— one with descriptions only, one with mock-generated Spanish utterances —
and runs five paraphrased Spanish prompts through each. Asserts the
enriched corpus's top-1 accuracy is **strictly greater** than the baseline
when baseline < 1.0. This proves the architecture (enrichment writes
intents the matcher then picks up); it does not validate any specific LLM.

### Empirical test (live LLM, gated)

The same test family is marked `@pytest.mark.enrichment`. Operators can
run `pytest -m enrichment` against a real LLM-enriched corpus to measure
the SkillRouter paper's hypothesis end-to-end via `cos-routing-benchmark`.

### Acceptance criteria

1. `.venv/bin/python -m pytest tests/unit/test_skill_description_enricher.py -v -m "not enrichment"` → all unmarked tests pass.
2. `scripts/cos-skill-description-enrich --dry-run --skills product-answer` runs without error and prints proposed enrichments.
3. `scripts/cos-adr-implementation-audit.py --strict` → 0 overclaims (all `implementation_files:` exist on disk).
4. Idempotency: rerunning enrich on an already-enriched skill produces zero net writes (verified by file-mtime assertion in the cost-cap test and by `already_enriched` skip).
5. The "enriched corpus improves routing" test passes with mock dispatch.


```bash
python3 -m pytest tests/unit -q
```
## Consequences

### Positive

- Closes the SkillRouter description-only accuracy gap.
- The corpus enrichment is one-shot, version-controlled (git diff is
  reviewable), and reversible.
- No new dependencies — uses existing `lib/dispatch.py`.
- Operator can curate intents incrementally; auto-generated tag prevents
  the enricher from clobbering manual work.

### Negative / trade-offs

- One-time cost (~$2 for the full catalog) and ~10 minutes wall-clock.
- Quality of utterances is provider-dependent; the strict-JSON contract +
  audit trail mitigate silent corruption but cannot guarantee semantic
  fidelity.
- ~385 SKILL.md files will gain 12 lines of frontmatter each — diff
  reviewers should expect a large but mechanically uniform PR when the
  enricher is run on the full catalog.

## Operational notes

- Re-run after large catalog changes (skill additions/renames) so new
  skills get coverage. The audit trail makes it easy to detect coverage
  gaps (`grep -v success.*true .cognitive-os/metrics/skill-enrichment.jsonl`).
- If `COS_DISABLE_ENRICHMENT=1` is set in CI, the tool is a no-op — safe
  to wire into automated jobs that should not spend tokens.
