# ADR-030 — Auto-trigger session-wrapup (Q1 prompt-match + Q2 commit banner)

**Status**: Accepted — implemented (Q1 trigger + Q2 commit banner shipped)
**Date**: 2026-04-20
**Related**: ADR-028 (observability), session-init nudge in commit `3e24256`

## Context

Today the orchestrator has 7 session lifecycle skills (`/session-wrapup`, `/session-backlog`, `/session-report-executive`, `/resume-tasks`, `/session-manager`, `/sdd-resume`, `/recall-search`). They are powerful but require **manual invocation**. Evidence from the 2026-04-20 session: the user had to repeatedly ask "guardá el summary" and explicitly point out that these skills existed — the orchestrator defaulted to manual ad-hoc saves.

Goal: make the common case automatic without making it rigid.

## Decision

Wire two opt-in auto-trigger mechanisms. Each is advisory-only: the skill is **suggested** to the orchestrator, never forced. Neither replaces the manual invocation path.

### Q1 — UserPromptSubmit regex trigger (suggest /session-wrapup)

**Mechanism.** New hook `hooks/session-wrapup-trigger.sh` registered under `UserPromptSubmit`. On every user prompt:

1. Read stdin JSON; extract `user_prompt`.
2. Match case-insensitive regex against closure phrases:
   ```
   (cerr(a|á|emos|emosla) (la )?sesi[oó]n|
    session (close|end|wrap ?up)|
    terminamos|
    listo[, ]? (ce|cerr)|
    cerrar sesion)
   ```
3. On match, emit `hookSpecificOutput.additionalContext`:
   ```
   AUTO-TRIGGER: user requested session close.
   Invoke /session-wrapup before any other action. Do not skip.
   ```
4. No match → exit 0 silently.

The orchestrator honours the auto-trigger per a rule added to `templates/agent-preamble.md`:

> When the agent preamble / hook additionalContext contains a line beginning
> `AUTO-TRIGGER:`, the orchestrator MUST execute the named skill before any
> other tool call, unless the user explicitly countermanded it in the same
> message.

### Q2 — Post-commit banner (deferred /session-wrapup reminder)

**Mechanism.** Git `post-commit` hook writes a breadcrumb file. Next `SessionStart` surfaces it.

- `.githooks/post-commit` (new) OR extension to the existing one: after every commit, append one line to `.cognitive-os/runtime/commit-nudge` with `timestamp | sha | subject`.
- `hooks/session-init.sh` (extend existing nudge block): if `commit-nudge` exists AND any commit in it is from the current day, append to the SessionStart advisory:
  ```
  📋 Commits since last wrapup: N (latest: <sha> <subject>).
     Invoke /session-wrapup before closing to archive today's work.
  ```
- `/session-wrapup` skill (when invoked) deletes `commit-nudge` as its last step, so the banner only fires when there's genuinely un-archived work.

**Q2 is NOT a direct auto-trigger** (the orchestrator can't invoke a skill mid-commit). It is a *delayed* trigger: the banner at the next session start nudges the orchestrator / user to archive. This is the arch-limit: git hooks run outside Claude Code's context and cannot call skills.

### Rejected: Q2-hard (git-hook directly invokes skill)

Requires Claude Code to be running, expose an IPC endpoint, and accept external skill invocations. All 3 are architectural lifts (new daemon, API surface, auth). Not worth the complexity for a convenience feature. Documented so future proposers see the prior analysis.

## Testing strategy

### Q1 testable path (pytest)

- **Unit** (`tests/hooks/test_session_wrapup_trigger.py`):
  - Pipe JSON `{"user_prompt":"cerremos sesión"}` → assert stdout contains `AUTO-TRIGGER:` + `/session-wrapup`
  - Pipe `{"user_prompt":"hola"}` → assert stdout does NOT contain `AUTO-TRIGGER`
  - Test 5+ regex variants (cerrá la sesión / session wrap up / terminamos / etc.)
  - Test edge: malformed JSON → exit 0, no output

The part that is **not unit-testable** is whether the orchestrator actually honours the auto-trigger — that is a runtime behaviour only observable in a real Claude Code session.

### Q1 honour test — log-then-reconcile

Claude Code does NOT expose raw conversation transcripts to hooks.
Prior ADR proposal to read `.cognitive-os/sessions/*/transcript.jsonl`
was incorrect — those files do not exist.

Correct approach (implemented):

1. `hooks/session-wrapup-trigger.sh` logs every AUTO-TRIGGER emission
   to `.cognitive-os/metrics/auto-trigger-events.jsonl` (MetricEvent
   source=`session-wrapup-trigger`, event_type=`auto_trigger.emitted`).

2. `hooks/skill-invocation-logger.sh` (PostToolUse:Skill) logs every
   Skill tool call to `.cognitive-os/metrics/skill-invocations.jsonl`
   (MetricEvent source=`skill-invocation-logger`, event_type=`skill.invoked`).

3. `tests/integration/test_auto_trigger_honoured.py` pairs emissions
   with subsequent invocations within a 60 s window. Computes
   compliance rate (% of emissions honoured). Non-compliant events
   are listed for manual inspection.

Limitations (honest):
- Starts accumulating data only after the two hooks are wired.
  Historical sessions are not testable.
- "User countermanded" is not distinguishable from "orchestrator
  ignored" — both look like non-compliance at this layer. Low
  severity: the preamble rule states countermand is a legitimate
  exception.
- Test is metric-level (reports rate), not per-session pass/fail.
  Below 80% rate logs a warning to engram but does not fail the
  test today. Hard-fail threshold to be tuned after 2 weeks of data.

### Q2 testable path (end-to-end in tmp_path)

- **Integration** (`tests/integration/test_post_commit_banner.py`):
  1. Set up a tmp git repo: `git init tmp_path/repo`; copy `hooks/session-init.sh` + the post-commit hook into place; symlink `.git/hooks/post-commit`.
  2. Make a commit: `git -C <tmp> commit --allow-empty -m "test"`.
  3. Assert `.cognitive-os/runtime/commit-nudge` exists and contains the commit SHA + subject.
  4. Invoke `session-init.sh` against this tmp repo; capture stderr; assert "Commits since last wrapup" appears with the recorded SHA.
  5. Simulate `/session-wrapup` by deleting `commit-nudge`; invoke session-init again; assert no banner.
  6. Edge: stale banner (> 24h old) should NOT trigger (prevents perpetual nudging). Fake file mtime via `os.utime`.

- Each step is plain subprocess.run; no Claude Code required. Fully hermetic in tmp_path.

## Rollback

- `session-wrapup-trigger.sh`: unregister from profile scripts → no trigger fires.
- `commit-nudge` banner: delete the marker file + remove the `post-commit` append line.
- Auto-trigger preamble rule: remove the paragraph from `templates/agent-preamble.md`.

All 3 rollbacks are independent; pick any.

## Residual risks

- **R-A (over-trigger)**: regex in Q1 fires on false positives (e.g. "cerremos **este file**"). Mitigation: require whole-word match + negative lookahead for common false-positives. Measure false-positive rate over 2 weeks; tighten if > 5%.
- **R-B (banner noise)**: if user commits many times a day without wrapping up, the banner nags. Mitigation: 24 h staleness cap (documented in Q2 testable path step 6).
- **R-C (orchestrator disobedience)**: orchestrator ignores `AUTO-TRIGGER:` advisory. Fails open — worst case the skill just isn't invoked, same as today. Detected via integration test above.
