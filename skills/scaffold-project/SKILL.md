# Scaffold Project

Create or update the `.claude/` directory structure for a project. Generates project-specific rules, skills, and hooks based on `.cognitive-os/detected-stack.json`.

## What It Does

Writes project-specific files to `{project}/.claude/`:
- **Rules**: architecture patterns, constitutional gates, service configs
- **Skills**: SRE agent configured for this project's containers, health checks
- **Hooks**: production URL blocking, auto-test with the project's test commands

## Invocation

```
/scaffold-project
```

Run from the project root after `/detect-stack` has produced `.cognitive-os/detected-stack.json`.

## Precondition

`.cognitive-os/detected-stack.json` must exist. If it does not, stop and instruct the user to run `/detect-stack` first (or `/generate-config` to also update `cognitive-os.yaml`).

## Execution Protocol

### Step 1: Read Stack Data

Read `.cognitive-os/detected-stack.json`.

### Step 2: Create Directory Structure

Create these directories if they do not exist:

```
.claude/
  rules/
  skills/
  hooks/
```

### Step 3: Generate .claude/rules/architecture.md

Based on detected services and language:

- For multi-service projects: document the communication flow between detected services
- For monoliths: document the primary module structure
- Include service names, ports, and directories from `detected-stack.json`

Template structure:
```markdown
# Architecture

## Services
{list of services with name, path, port, language}

## Communication Flow
{inferred or placeholder — to be filled by the team}

## Layer Conventions
{language-specific conventions, e.g. for Go: internal/, cmd/, pkg/}
```

### Step 4: Generate .claude/rules/constitutional-gates.md

Based on `project.type` from `detected-stack.json`:

| Project Type | Gates |
|-------------|-------|
| `fintech` | idempotency for all financial ops, audit trails mandatory, mock-before-integrate for payment providers (example) |
| `ecommerce` | inventory consistency checks, payment mock-first, order idempotency |
| `healthcare` | HIPAA compliance reminders, PHI must be encrypted at rest, audit log every data access |
| `saas` | multi-tenant data isolation, subscription state consistency |
| `webapp` (default) | test-before-merge, secrets in env vars only, backward-compatible APIs |

### Step 5: Generate .claude/rules/services-config.md

Using detected infrastructure from `detected-stack.json`:

```markdown
# Services Configuration

## Port Mapping
{service}: {port}

## Required Environment Variables
{inferred from detected auth/database/cache/messaging}

## Local Development Credentials
(These are local-only defaults — never use in production)
{common local dev defaults for detected services}
```

### Step 6: Generate .claude/skills/sre-agent-config/SKILL.md

Write an SRE agent overlay for this specific project:

```markdown
# SRE Agent Config — {project_name}

## Container-to-Directory Mapping
{container_name} → {source_path}

## Port-to-Service Mapping
{port} → {service_name}

## Health Check Endpoints
{service_name}: {inferred_health_path}
```

Use `/health` or `/api/health` as the default health check path unless a specific path is detectable from the codebase.

### Step 7: Generate .claude/skills/health-check/SKILL.md

Write a health check skill with actual service endpoints:

```markdown
# Health Check — {project_name}

## Services

{for each service}
- **{service_name}**: GET http://localhost:{port}/health — expect 200
{end for}

## Dependency Chain
{inferred order: auth → database → cache → api services}
```

### Step 8: Generate .claude/hooks/block-prod-urls.sh

Detect production URLs from `.env` files or common config patterns, then generate a hook that warns when production URLs appear in local dev contexts:

```bash
#!/usr/bin/env bash
# Generated by scaffold-project for {project_name}
# Warns when production URLs are used in local development

PROD_PATTERNS=(
  {detected_prod_url_patterns}
)

# ... hook logic using content-policy pattern ...
```

If no production URLs are detectable, generate a template with placeholder patterns and a comment instructing the developer to fill them in.

Mark the hook as executable: `chmod +x .claude/hooks/block-prod-urls.sh`

### Step 9: Update Workflow Config

Write or update `.cognitive-os/workflows/config/services.yaml` with detected services.

### Step 10: Report

```
== Scaffold Project Complete ==

Created .claude/ structure:
  .claude/rules/architecture.md
  .claude/rules/constitutional-gates.md
  .claude/rules/services-config.md
  .claude/skills/sre-agent-config/SKILL.md
  .claude/skills/health-check/SKILL.md
  .claude/hooks/block-prod-urls.sh
  .cognitive-os/workflows/config/services.yaml

Next steps:
  1. Review generated files and adjust as needed
  2. Run /cognitive-os-test to verify configuration
  3. Run /sre-agent to verify service monitoring
```

## Idempotency

Re-running `/scaffold-project` overwrites generated files after prompting for confirmation. Manual customizations should be made to files AFTER scaffold-project generates them; note that re-running will overwrite those customizations.

## Output Contract

Writes files to `.claude/` and `.cognitive-os/workflows/config/`. These are the sole outputs. Does not modify `cognitive-os.yaml` (that is `/generate-config`'s responsibility) or `detected-stack.json` (that is `/detect-stack`'s responsibility).
