# Verify the Public Release Yourself

> Anti-FUD toolkit. This document is the long-form companion to
> [TRANSPARENCY.md §6](../../TRANSPARENCY.md#6-verify-it-yourself-anti-fud-toolkit).
> Every command runs against a fresh clone with no privileged input. If a
> result diverges from what we claim here, file an issue.

This page exists because the repository was rewritten on 2026-05-08 and
the license moved from Apache-2.0 to FSL-1.1-MIT before public flip.
Both events change what a skeptic sees in `git log` and in `LICENSE`.
Rather than ask anyone to take our word, we publish the verification
recipe.

---

## 1. SBOM hash verification

The SBOM ships at the repo root as `sbom.json` (CycloneDX 1.6 JSON
generated by `syft 1.44.0`). Per
[`docs/security/supply-chain.md`](supply-chain.md) §1.2, the committed
SBOM has a published SHA-256.

```
shasum -a 256 sbom.json
jq -r '"\(.bomFormat) \(.specVersion)"' sbom.json
# Expect: CycloneDX 1.6
```

Note from supply-chain.md §1.2: `syft` writes a fresh `serialNumber` and
timestamp into every SBOM, so byte-identical reproduction across two
runs is not expected. To verify two SBOMs describe the same dependency
closure, compare the deduplicated `(purl, version)` tuple set:

```
jq '[.components[]|.purl]|unique' sbom.json
```

## 2. Signed-tag verification

Per [`docs/security/release-signing.md`](release-signing.md):

- Pre-`v1.0.0` tags are unsigned by disclosed gap. Running
  `git tag -v vX.Y.Z` against any of those tags returns
  `error: no signature found`. This is documented, not a tampering
  signal.
- The first tag flagged `public-release: true` (the first `v1.0.0+` tag)
  ships signed and verifies cleanly:

```
git fetch --tags
git tag -v v1.0.0
```

If the signature fails to verify on a release flagged as public, treat
the release as compromised and follow [TRANSPARENCY.md §7](../../TRANSPARENCY.md#7-contact--responsible-disclosure).

## 3. Manifest-snapshot byte-diff

The pre-rewrite policy was frozen at
[`docs/history/manifest-snapshot-2026-05-07.yaml`](../history/manifest-snapshot-2026-05-07.yaml).
The live manifest is at
[`manifests/history-sanitization.yaml`](../../manifests/history-sanitization.yaml).

```
diff manifests/history-sanitization.yaml docs/history/manifest-snapshot-2026-05-07.yaml
```

The live manifest is a **superset** of the snapshot, not a different
ruleset. The differences are rules added between the snapshot date
(2026-05-07) and the rewrite execution (2026-05-08): `operator-name`,
`historical-fixture-home-paths`, the consumer-codename slots
(`consumer-codename-a..c`), and the consumer-service slots
(`consumer-service-name`, `-2`, `-3`, `-4`, `-5`).

Every rule still has a documented `rationale:` field — that is the
governance contract. If the live manifest contains a rule with no
rationale, or contains a rule whose pattern would also match the
`preserve` block, the canonical primitive refuses to execute (ADR-218
"Hard rules"). Independently audit:

```
python3 -c "import yaml; m=yaml.safe_load(open('manifests/history-sanitization.yaml')); [print(r['id'], '-', r.get('rationale','MISSING-RATIONALE')[:80]) for r in m['rules']]"
```

## 4. SHA-inventory sanity check

The pre-rewrite SHA inventory was published at
[`docs/history/pre-sanitization-sha-inventory-2026-05-07.txt`](../history/pre-sanitization-sha-inventory-2026-05-07.txt)
(1,775 commits, one line per commit, format `SHA AUTHOR-DATE SUBJECT`).

```
shasum -a 256 docs/history/pre-sanitization-sha-inventory-2026-05-07.txt
# Expect: 923170ead7ef9fd7c072089a699f867e111ead1be38a84ce41eb1a4023104997

wc -l docs/history/pre-sanitization-sha-inventory-2026-05-07.txt
# Expect: 1775
```

Now the proof-of-rewrite. Every SHA listed in the inventory must NOT be
reachable from the current `origin/main` — that non-reachability is
the cryptographic signature of the rewrite:

```
# Sample 20 random SHAs from the inventory and confirm they are unreachable.
shuf -n 20 docs/history/pre-sanitization-sha-inventory-2026-05-07.txt | \
  awk '{print $1}' | \
  while read sha; do
    if git cat-file -e "$sha" 2>/dev/null; then
      echo "REACHABLE (unexpected): $sha"
    else
      echo "missing (expected):     $sha"
    fi
  done
```

Expect: every line prints `missing (expected)`. If any pre-rewrite SHA
is reachable from `origin/main`, the rewrite was incomplete and the
operator owes a public explanation.

The complementary direction: every commit reachable from `origin/main`
is a *post-rewrite* SHA — it descends from the post-rewrite head
`db846adb6290456b431bfc191b08543f56a2e8d7`. The inventory SHAs and the
current SHAs partition cleanly: no overlap, by construction.

> **Why this is the proof.** A rewrite that genuinely scrubbed historical
> blob content cannot leave the original SHAs reachable: every commit
> re-hashes because every tree re-hashes because every modified blob
> re-hashed. If old SHAs are reachable on `origin/main`, either no
> rewrite happened or the rewrite was partial. The non-reachability
> property is binary and easy to test.

## 5. License preservation check

The license transition Apache-2.0 → FSL-1.1-MIT is public history. The
sanitization manifest's `preserve.license-transition` rule pins the
following strings as not-rewritable: `Apache License`, `Apache 2.0`,
`Apache-2.0`, `FSL-1.1`, `FSL-1.1-MIT`, `Functional Source License`.

```
# 1. Both Apache and FSL strings are present in the public artifacts.
grep -E 'Apache|FSL-1\.1-MIT' LICENSE NOTICE docs/legal/license-faq.md | wc -l
# Expect: 40+ matches.

# 2. The LICENSE file is the FSL-1.1-MIT text.
head -5 LICENSE
# Expect: starts with the FSL-1.1-MIT preamble.

# 3. NOTICE retains Apache attributions for third-party Apache-licensed
#    code we depend on.
grep -c 'Apache License, Version 2.0' NOTICE
# Expect: nonzero (counts third-party Apache attributions).

# 4. The license FAQ documents the transition explicitly.
grep -E 'Apache.*FSL|FSL.*Apache' docs/legal/license-faq.md
# Expect: text describing the move.
```

If any of these fail, the rewrite either broke its preserve contract or
the `LICENSE` file was tampered with after publication. Either is a
disclosure-grade event.

## 6. Putting it together

A complete pass:

```
git clone https://github.com/luum-home/luum-cognitive-os && cd luum-cognitive-os

# (1) SBOM exists and parses
test -f sbom.json && jq -r '"\(.bomFormat) \(.specVersion)"' sbom.json

# (2) Tag verification (post-v1.0.0 only)
git fetch --tags && git tag -v v1.0.0 || echo "expected pre-v1.0.0 to be unsigned"

# (3) Manifest snapshot diff is bounded to documented additions
diff manifests/history-sanitization.yaml docs/history/manifest-snapshot-2026-05-07.yaml | head -50

# (4) Pre-rewrite SHAs are unreachable from origin/main
shasum -a 256 docs/history/pre-sanitization-sha-inventory-2026-05-07.txt
shuf -n 20 docs/history/pre-sanitization-sha-inventory-2026-05-07.txt | \
  awk '{print $1}' | \
  while read sha; do git cat-file -e "$sha" 2>/dev/null && echo REACHABLE || echo missing; done

# (5) License strings preserved
grep -E 'Apache|FSL-1\.1-MIT' LICENSE NOTICE docs/legal/license-faq.md | wc -l
```

If all five pass, you have independently confirmed:

- the project ships a CycloneDX 1.6 SBOM;
- (post-v1.0.0) the release tag's signature matches the maintainer's
  public verification material;
- the live sanitization policy is a documented superset of the frozen
  pre-rewrite snapshot, not a replacement;
- the rewrite actually happened (pre-rewrite SHAs are unreachable);
- the license transition was preserved through the rewrite.

Nothing in this list requires trusting Luum or the operator.
